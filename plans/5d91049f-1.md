## Current state

The canonical conditional final theorem
`BHKS.factorFast_terminates`
([HexBerlekampZassenhausMathlib/Recovery.lean:5053](../HexBerlekampZassenhausMathlib/Recovery.lean#L5053))
landed via #6418 / progress
[`progress/20260603T034948Z_1b42c33c.md`](../progress/20260603T034948Z_1b42c33c.md).
Its signature is

```lean
theorem factorFast_terminates
    (f : Hex.ZPoly) (primeData : Hex.PrimeChoiceData)
    (rows_pos : HasPositiveDimension
      (Hex.normalizeForFactor f).squareFreeCore
      (factorFastCapLiftData f primeData))
    (trueSupports : Set (Set (Fin ...factorCount)))
    (capInputs :
      FactorFastCapSeparationInputs f primeData rows_pos trueSupports)
    (recoveryInputs :
      CanonicalRecoveryTailInputs f primeData rows_pos trueSupports) :
    Hex.factorFast f ≠ none
```

so its applicability rests on supplying the two closed packages
`FactorFastCapSeparationInputs` and `CanonicalRecoveryTailInputs` at the
actual `factorFast` cap.

Each of those packages now has a closed producer in flight:

- `BHKS.FactorFastCapSeparationInputs.ofBridgeDataPointwiseAuxiliaryBoundsAndPaperThreshold`
  ([HexBerlekampZassenhausMathlib/Recovery.lean]) — the cap-side wrapper
  landed by partial-#6445 PR #6453 — accepts a paper-threshold bound
  `hpaper` plus pointwise vector/correction bounds, dispatching the strict
  cap-arithmetic comparison via
  `bhksPaperThresholdReal_chain_lt_p_pow_kLocalFactorDegree`.
  Issue #6452 (blocked on partial-#6445 landing) discharges `hpaper` for
  the canonical bad-vector bridge, producing a closed
  `FactorFastCapSeparationInputs` package from a `ForwardRecoveryInputs`-
  compatible canonical bridge.
- `BHKS.CanonicalRecoveryTailInputs.ofCanonicalSupportRepresentations`
  ([HexBerlekampZassenhausMathlib/Recovery.lean:3794](../HexBerlekampZassenhausMathlib/Recovery.lean#L3794))
  consumes (i) a `ForwardRecoveryInputs.ExpectedTrueFactors` package and
  (ii) per-canonical-class `RepresentsIntegerFactorAtLift` certificates
  against the canonical `liftedFactorSubsetsOfSupports` array.
  Issues #6448 and #6450 produce (i) and (ii) respectively from a
  successful `Hex.bhksIndicatorCandidates? f d expectedIndicators = some
  expectedFactors`, which is exactly the executable witness carried by
  `ForwardRecoveryInputs f d`
  ([HexBerlekampZassenhausMathlib/Recovery.lean:612](../HexBerlekampZassenhausMathlib/Recovery.lean#L612)).

What is missing is the final composition wrapper that consumes a single
`ForwardRecoveryInputs f (factorFastCapLiftData f primeData)` (plus the
canonical `trueSupports`, `rows_pos`, and BHKS dominance ingredients) and
returns `Hex.factorFast f ≠ none`. The "Next step" of
[`progress/20260603T034948Z_1b42c33c.md`](../progress/20260603T034948Z_1b42c33c.md)
flags exactly this gap:

> Construct concrete (per-input or universal) producers for
> `FactorFastCapSeparationInputs` and `CanonicalRecoveryTailInputs` for
> non-pathological inputs, so that `factorFast_terminates` becomes
> applicable without further hypotheses on the input polynomial beyond
> the `choosePrimeData?` success witness.

The intent is **not** to remove the `choosePrimeData?` hypothesis (which
remains required per the `finitePrimeSearchNoneQuadratic` counterexample
documented on `factorFast_terminates`), but to collapse the two-package
consumer surface into a single `ForwardRecoveryInputs`-keyed wrapper that
matches the existing forward-recovery surface used elsewhere in the
library.

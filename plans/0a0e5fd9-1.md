## Current state

`FpPoly.mulCoeffTerm` and `FpPoly.mulCoeffSum` are public definitions in
[HexPolyFp/Basic.lean:1551-1556](../HexPolyFp/Basic.lean#L1551), sitting
beside the ordinary `FpPoly` constructors and operation laws:

```lean
/-- The `i`th schoolbook contribution to coefficient `n` of `f * g`. -/
def mulCoeffTerm (f g : FpPoly p) (n i : Nat) : ZMod64 p :=
  if n < i then 0 else f.coeff i * g.coeff (n - i)

/-- The executable schoolbook coefficient sum matching `FpPoly` multiplication. -/
def mulCoeffSum (f g : FpPoly p) (n : Nat) : ZMod64 p :=
  (List.range f.size).foldl (fun acc i => acc + mulCoeffTerm f g n i) 0
```

Every theorem building on them in the file (`coeff_mul_fold`,
`foldl_mulCoeffStep_*`, `mulCoeffTerm_*`, `fold_mulCoeff_*`,
`mulCoeffSum_eq_bound`, ~24 declarations across lines 1558-2150) is
`private`. The only outside-the-module consumers are six call sites in
[HexHensel/Linear.lean:277, 280, 416, 421, 482, 494](../HexHensel/Linear.lean)
that use `FpPoly.mulCoeffTerm` directly inside the linear Hensel lift
analytic obligation.

The public characterising lemma `coeff_mul` (built from this scaffolding)
is the intended downstream surface for users who only need coefficients
of `FpPoly` multiplication. The schoolbook helpers are proof-facing
infrastructure for Hensel lift proofs, not part of the computational
`FpPoly` API.

## Symptom

**Declared expectation.** From
[reports/hex-poly-fp-api-phase6-review.md §"4. Public schoolbook
coefficient helpers need an explicit proof-facing
home"](../reports/hex-poly-fp-api-phase6-review.md):

> `FpPoly.mulCoeffTerm` and `FpPoly.mulCoeffSum` are public and used
> outside `HexPolyFp`, especially by Hensel lift proofs. They are
> legitimate proof-facing infrastructure, but their current placement
> beside ordinary polynomial constructors makes them look like part of
> the computational API. They also duplicate the generic
> `DensePoly.mulCoeffSum` vocabulary, increasing the chance that
> downstream code unfolds the wrong layer.

**Observed result.** The two helpers are exposed in the public `FpPoly`
namespace with docstrings that describe the schoolbook role but do not
flag them as Hensel-proof-facing. A reader scanning the public API
surface would treat them as ordinary `FpPoly` multiplication
characterisations alongside `coeff_mul`.

**Root-cause hypothesis.** During Phase 5 the helpers were promoted
public because `HexHensel/Linear.lean` needed them directly. The
review (`#6022`-era audit) noted the placement but the follow-up was
not filed at the time; findings 1-3 from that review have been
addressed by the SquareFree refactor chain and `#6231` (neutral-laws
simp pass), leaving finding 4 outstanding.

## Deliverables

Pick **one** of the two routes from the review.

### Option A (preferred if low-friction): section-header documentation

1. Add a section header (or `/-! ... -/` block-doc) immediately above
   the `mulCoeffTerm` declaration at
   [HexPolyFp/Basic.lean:1550](../HexPolyFp/Basic.lean#L1550) that
   names the block as **proof-facing Hensel scaffolding** and explicitly
   directs ordinary `FpPoly` multiplication callers to the public
   `coeff_mul` lemma instead.
2. Expand the existing docstrings on `mulCoeffTerm` and `mulCoeffSum`
   to repeat the proof-facing-only contract in one short sentence each
   (so a hover over the declaration in the IDE conveys the warning).
3. No source change to consumers; no namespace move.

### Option B: namespace move to `FpPoly.Internal`

1. Wrap `mulCoeffTerm`, `mulCoeffSum`, and the cluster of private
   theorems that mention them in a `namespace FpPoly.Internal ... end`
   block (or rename to `FpPoly.Internal.mulCoeffTerm`,
   `FpPoly.Internal.mulCoeffSum`). Keep their definitional bodies and
   proofs unchanged.
2. Update the six `HexHensel/Linear.lean` call sites
   (lines 277, 280, 416, 421, 482, 494) to the new fully-qualified
   names or add a local `open FpPoly.Internal` where appropriate.
3. Preserve `FpPoly.coeff_mul` as the public characterising lemma for
   ordinary multiplication; do not move it.

Either route discharges the review finding. Option A is appropriate if
the namespace move would churn unrelated proof bodies; option B is
appropriate if `FpPoly.Internal` already exists or can be introduced
cheaply for future proof-facing helpers.

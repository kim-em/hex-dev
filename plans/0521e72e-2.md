## Context

The `fpPoly_dvd_trans` audit (#5082, closing audit comment on
2026-05-18) explicitly flagged this site as "a candidate for a
separate `fpPoly_mul_dvd_mul` extraction, but that is outside
the scope of the `fpPoly_dvd_trans` cluster". This planning
cycle promotes that out-of-scope observation to a tracked work
item.

The helper is parallel in framing to `fpPoly_dvd_trans`:

- Both helpers exist because `Hex.FpPoly p` uses
  `instDvdOfAddOfMul`, which Mathlib's general-purpose
  `dvd_trans` / `mul_dvd_mul` lemmas don't see through.
- Both discharge a small ring-style witness via
  `Hex.FpPoly.mul_assoc` (and, for this helper,
  `Hex.FpPoly.mul_comm`) — no other algebraic moves.
- Both are `private theorem`s in `Basic.lean` at the FpPoly
  helper band.

This is a single-site extraction. The collapse is ~14 net lines
(21 → 1 term-mode line + the helper body, which is ~7 lines).
The value is readability — the inline calc chain interrupts the
flow of the larger 80-line
`quadraticMultifactorCoprimeSplits_of_factorProduct_no_squared`
recursive step, where the higher-level structure (Step 1: square
divides X; Step 2-7: rawGcd is a nonzero constant; final scale
discharge) is what the proof is actually doing.

The audit pattern's standard follow-up sequence applies: a
post-extraction audit issue (analogous to #5082 for
`fpPoly_dvd_trans`, #5085 for `fpPoly_isZero_false_of_dvd_of_isZero_false`)
can be planned as a separate review work item after this lands.
Do **not** file the post-extraction audit issue as part of this
work — it belongs to a separate planner cycle.

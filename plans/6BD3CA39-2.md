`HexPolyFp/Basic.lean` defines `FpPoly.linearPow` and its surrounding
arithmetic lemmas (lines 3554–3624):

```lean
def linearPow (f : FpPoly p) : Nat → FpPoly p
  | 0 => 1
  | n + 1 => linearPow f n * f

@[simp] theorem linearPow_zero (f : FpPoly p) : linearPow f 0 = 1 := rfl

theorem linearPow_succ (f : FpPoly p) (n : Nat) :
    linearPow f (n + 1) = linearPow f n * f := rfl

theorem linearPow_succ_left ...
theorem linearPow_add ...
theorem linearPow_iterate_mul ...
theorem eval_linearPow ...
theorem linearPow_monomial ...
```

The `linearPow_succ` / `linearPow_succ_left` / `linearPow_add` /
`linearPow_iterate_mul` / `eval_linearPow` cluster lacks docstrings on
several public theorems, and there are no small-exponent or scalar-base
aliases. In particular, the `n = 1`, `f = FpPoly.C c`, and
`f = FpPoly.X` corners — which downstream Rabin/Frobenius proofs reach
through hand-written rewrites — are not exposed.

This was identified as the second follow-up slice in the merged Phase 6
review of `HexPolyFp` (issue #5131; review comment on PR #5136). The
other three review slices — coefficient lemmas for `FpPoly.C`/`FpPoly.X`,
quotient `% g`-oriented aliases, and a quotient/Frobenius bridge — are
deliberately not in scope here.

## Deliverables

1. In `HexPolyFp/Basic.lean`, immediately after the existing
   `linearPow_succ_left` cluster (after line ~3577) and before
   `linearPow_add` (line 3579), add the small-exponent helper:
   - `theorem FpPoly.linearPow_one (f : FpPoly p) :
       linearPow f 1 = f` proved by `linearPow_succ` + `one_mul` (the
     existing `@[simp] theorem one_mul` at line 1503 applies).

2. Immediately after `linearPow_monomial` (after line ~3624), add the
   two scalar-base aliases:
   - `theorem FpPoly.linearPow_X (n : Nat) :
       linearPow (FpPoly.X : FpPoly p) n =
         DensePoly.monomial n (1 : ZMod64 p)`
     proved by unfolding `FpPoly.X` and reusing the existing
     `linearPow_monomial` with `k = 1`.
   - `theorem FpPoly.linearPow_C (c : ZMod64 p) (n : Nat) :
       linearPow (FpPoly.C c) n = FpPoly.C (c ^ n)` (or, if `^` is not
     available at this level, the `Lean.Grind.Semiring.pow` form already
     used by `eval_linearPow` at line 3604). The proof is induction on
     `n` using `linearPow_succ` + the existing `FpPoly` multiplication
     of constants. If the cleanest statement exposes pow on `ZMod64 p`
     rather than `c ^ n`, prefer the form that closes with
     `Lean.Grind.Semiring.pow_succ` / `pow_zero` to mirror
     `eval_linearPow`.

3. Add a one-line docstring to each of the existing public theorems
   that currently lack one in this cluster: `linearPow_succ`,
   `linearPow_succ_left`, `linearPow_add`. Do not change their
   statements or proofs. (`linearPow_zero` and `linearPow_iterate_mul`
   already have appropriate context; do not touch their proofs.)

4. Do not add `@[simp]` to any of the new theorems. The review's
   simp-risk note specifically calls out that `linearPow` lemmas
   expanding to multi-step rewrites are not safe simp candidates. The
   only existing simp lemma in this cluster, `linearPow_zero`, stays
   as-is.

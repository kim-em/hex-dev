The library's executable and theorem surface lives in
`HexConway/Basic.lean` (3425 lines) and the top-level re-export
`HexConway.lean`. Public surface includes:

- the table-lookup wrappers `luebeckConwayCoeffs?`,
  `luebeckConwayPolynomialOfCoeffs`, `luebeckConwayPolynomial?`
  (lines 23, 63, 89);
- per-committed-entry definitions `luebeckConwayPolynomial_p_n`
  (36 definitions);
- per-entry `_monic` theorems (36 occurrences) and `_irreducible`
  theorems;
- six `ZMod64.Bounds` instances for the supported small primes
  and six `ZMod64.PrimeModulus` instances;
- per-entry `Berlekamp.IrreducibilityCertificate` data and
  certificate-check theorems for the irreducibility evidence.

A first scan suggests at least one API inconsistency that a Phase 6
review would surface and either justify or file a follow-up for:
`_degree_pos` theorems exist only for the six `p = 2` entries
(`luebeckConwayPolynomial_2_1` ... `_2_6`), while the 30 other
committed entries (across `p = 3, 5, 7, 11, 13`) carry `_monic` and
`_irreducible` without a parallel `_degree_pos`. The review should
either justify the asymmetry (e.g. the downstream `HexGFq` consumer
needs `degree_pos` only on `p = 2`) or recommend a follow-up to
close it.

The downstream consumer is `HexGFq/Basic.lean` (imports `HexConway`).
The review must check whether the consumer can use the committed
table API through named theorems without unfolding
`luebeckConwayPolynomial?`, `luebeckConwayCoeffs?`, or per-entry
definitions, and whether it can recover the irreducibility witness
from public theorems rather than the underlying certificate data
structure.

This is a review-only issue. It must not edit Lean source.

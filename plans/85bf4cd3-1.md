recommended follow-ups. Recommendation #2 was already landed/queued as

Today:

- `HexHensel/QuadraticMultifactor.lean:922` exposes
  `quadraticMultifactorLiftInvariant_of_factorsModP`: a public
  constructor for `QuadraticMultifactorLiftInvariant` from mod-`p`
  boundary facts (monicity, lifted ordered product congruence mod `p`,
  per-split normalised-XGCD coprimality, nonempty list).
- `HexHensel/Multifactor.lean:131` defines the parallel linear-path
  invariant `MultifactorLiftInvariant` as a Prop with four conjuncts
  (loop invariant at `n = 1`, per-iteration step-degree invariant,
  per-iteration step-Bezout congruence, recursive tail). There is no
  analogous `multifactorLiftInvariant_of_factorsModP` constructor.
- `HexHensel/Multifactor.lean:308` exposes `multifactorLift_spec`,
  which consumes `MultifactorLiftInvariant`. There is no
  `_of_choosePrimeData` wrapper.
- The current downstream BZ consumer
  (`HexBerlekampZassenhaus/Basic.lean:2418`) uses
  `QuadraticMultifactorLiftInvariant` only. `MultifactorLiftInvariant`
  (the linear flavour) is referenced by
  `HexHenselMathlib/Correctness.lean:435`
  (`multifactorLift_eq_multifactorLiftQuadratic`, currently `sorry`)
  but has no executable-side caller.

The review's "or" branch is genuinely viable: if the linear lifter is
not intended to grow a public consumer surface, the right move is to
document that and leave `MultifactorLiftInvariant` proof-internal.

## Deliverables

Pick one of the two branches below and execute it. The choice is
worker-side based on a brief look at consumers; both branches are
acceptable as long as the rationale is recorded in the docstring.

### Branch A — add the linear-path constructor

In `HexHensel/Multifactor.lean`, add a public theorem mirroring the
quadratic constructor:

```lean
theorem multifactorLiftInvariant_of_factorsModP
    (p k : Nat) [ZMod64.Bounds p] [ZMod64.PrimeModulus p]
    (f : ZPoly) (factors : List (FpPoly p))
    (hp : 1 < p)
    (hk : 1 ≤ k)
    (hf_monic : DensePoly.Monic f)
    (hfactors_monic : ∀ g ∈ factors, DensePoly.Monic g)
    (hproduct_mod_p :
      ZPoly.congr
        (Array.polyProduct ((factors.map FpPoly.liftToZ).toArray))
        f p)
    (hcoprime : ??? -- per-split normalised-XGCD coprimality, see below)
    (hnonempty : factors ≠ []) :
    MultifactorLiftInvariant p k f (factors.map FpPoly.liftToZ)
```

Argument order, parameter names, hypothesis bundling, and docstring
style should match `quadraticMultifactorLiftInvariant_of_factorsModP`
exactly so the two read as a paired API. Reuse the existing
`QuadraticMultifactorCoprimeSplits` shape if it transfers cleanly to
the linear case, or define a private `LinearMultifactorCoprimeSplits`
analogue with the same recursive head/tail pattern. Do not silently
strengthen the hypotheses.

The proof should be structurally parallel to the quadratic one:
induct on `factors`; in the `g :: h :: tail` arm,

1. assemble `hprod_split`, `hcoprime_split`, `hbezout`, `hg_monic`
   from the supplied mod-`p` facts;
2. build the four conjuncts of `MultifactorLiftInvariant` for the
   non-singleton case — the `LinearLiftLoopInvariant` at `n = 1`, the
   per-iteration step-degree invariant, the per-iteration step-Bezout
   congruence, and the recursive tail with `lifted.h` as the new
   target — using whatever existing linear-loop machinery already
   covers these in `HexHensel/Linear.lean`.

If a strict mirror of the quadratic proof requires new step-level
lemmas that do not already exist in `HexHensel/Linear.lean`, **stop**
and skip-replan the issue with a one-paragraph note describing the
missing substrate. Do not introduce that substrate inside this issue.

### Branch B — document as proof-internal

If a brief look at consumers confirms that `MultifactorLiftInvariant`
is not intended to be constructed by external callers (it is the
correctness substrate for `multifactorLift_spec` only, and downstream
code uses `QuadraticMultifactorLiftInvariant`), expand the existing
docstring on `MultifactorLiftInvariant` in
`HexHensel/Multifactor.lean:131` to say so explicitly. The new
language should:

- name `QuadraticMultifactorLiftInvariant` as the consumer-facing
  invariant for callers wanting a public Hensel-lift correctness
  surface;
- cite `quadraticMultifactorLiftInvariant_of_factorsModP` as the
  constructor that should be used in its place;
- reference `multifactorLift_eq_multifactorLiftQuadratic` in
  `HexHenselMathlib/Correctness.lean` as the route by which a
  downstream Mathlib-side caller of `multifactorLift` would obtain
  the linear-path lifted factors (modulo `p^k`) via the quadratic
  path's outputs.

Do not change executable behaviour, theorem signatures, or any other
file in this branch. The deliverable is the docstring.

In either branch:

- Do not introduce `sorry`, `axiom`, `native_decide`, `TODO`, or
  `FIXME`.
- Do not remove or rename any existing public theorem.
- Do not modify `MultifactorLiftInvariant`'s defining equation.

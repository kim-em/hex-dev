## Deliverables

1. **`berlekampFactor_factors_nodup` in
   `HexBerlekampMathlib/Basic.lean`.** A new theorem of the shape

   ```lean
   theorem berlekampFactor_factors_nodup
       (f : Hex.FpPoly p) (hmonic : Hex.DensePoly.Monic f)
       [Lean.Grind.Field (Hex.ZMod64 p)] [Hex.ZMod64.PrimeModulus p]
       (hsquareFree : Hex.DensePoly.gcd f (Hex.DensePoly.derivative f) = 1) :
       (Hex.Berlekamp.berlekampFactor f hmonic).factors.Nodup
   ```

   Place it adjacent to `irreducible_of_mem_berlekampFactor` (around
   `HexBerlekampMathlib/Basic.lean:284`). The natural proof route
   transports through `fpPolyEquiv`:

   - the Mathlib image of the factor product equals
     `toMathlibPolynomial f` (via `prod_berlekampFactor` plus the
     ring-hom action of `fpPolyEquiv`);
   - `toMathlibPolynomial f` inherits squarefreeness from
     `hsquareFree` (gcd-with-derivative criterion transports under
     `fpPolyEquiv`); a Mathlib `Squarefree`/`IsSquarefree` statement
     is fine here;
   - in a unique factorization domain, the factors of a squarefree
     element are pairwise non-associated;
   - the Berlekamp factors are individually monic (this monicness is
     not yet stated as a stand-alone lemma; if a clean Mathlib-free
     statement is too large for this issue, deliver it as a small
     auxiliary `berlekampFactorLoop_factors_monic` lemma in
     `HexBerlekamp/Factor.lean` proved by induction on `fuel`, with
     `splitWithWitnesses?` shown to produce monic split factors);
   - pairwise non-associated monics are pairwise distinct, so the
     Mathlib factor list is `Nodup`;
   - pull `Nodup` back through `fpPolyEquiv` injectivity.

   A different proof route is acceptable if cleaner, but document
   the chosen route in the theorem's docstring.

2. **`factorsModP_nodup_of_factorsModPBerlekampForm` in
   `HexBerlekampZassenhausMathlib/Basic.lean`.** A wrapper exposing
   the Nodup property from the existing `factorsModPBerlekampForm`
   invariant plus the `isGoodPrime`-derived squarefreeness:

   ```lean
   theorem factorsModP_nodup_of_factorsModPBerlekampForm
       (f : Hex.ZPoly) (data : Hex.PrimeChoiceData)
       (hform : Hex.factorsModPBerlekampForm f data)
       (hgood :
         letI := data.bounds
         Hex.isGoodPrime f data.p = true) :
       data.factorsModP.toList.Nodup
   ```

   The `isGoodPrime` hypothesis supplies the modular-image
   squarefreeness needed by deliverable (1); the
   `factorsModPBerlekampForm` invariant unfolds
   `data.factorsModP = (berlekampFactor _ _).factors.toArray`, so
   `.toList.Nodup` reduces to `(berlekampFactor _ _).factors.Nodup`
   via `Array.toList_toArray` (or whichever existing lemma chains
   `(_ : List _).toArray.toList = _`).

   Place this wrapper inside the same Mathlib-bridge namespace as
   `henselLiftData_liftedFactor_injective_of_choosePrimeData`
   (the file's existing top-level namespace).

Placement answers:

1. **SPEC §.** `SPEC/Libraries/hex-berlekamp-mathlib.md:1–10` —
   "hex-berlekamp-mathlib … Proves the full correctness of
   Berlekamp's algorithm and Rabin's test by transferring to
   `Polynomial (ZMod p)` and using Mathlib's Euclidean domain
   theory." The Nodup-of-factors property is part of that
   correctness picture (factors of a squarefree element are
   pairwise distinct) and the route goes through Mathlib's UFD /
   squarefree theory, so it belongs in the Mathlib-bridge library
   alongside `irreducible_of_mem_berlekampFactor`. The
   `choosePrimeData`-shaped wrapper belongs in
   `SPEC/Libraries/hex-berlekamp-zassenhaus-mathlib.md`'s scope
   (line 1: "All proofs about the Mathlib-free routines defined in
   hex-berlekamp-zassenhaus"), since it speaks about
   `Hex.PrimeChoiceData` and its `factorsModPBerlekampForm`
   invariant.
2. **Mathlib?** Yes for deliverable (1) — the natural route uses
   Mathlib `UniqueFactorizationMonoid` / `Squarefree` theory for
   `Polynomial (ZMod p)`. Deliverable (2) lives in
   `HexBerlekampZassenhausMathlib` because it consumes (1).
3. **Already in Mathlib?** The general fact "factors of a
   squarefree element in a UFD are pairwise non-associated" is in
   Mathlib (`UniqueFactorizationMonoid.squarefree_iff_nodup_factors`
   and surrounding lemmas); import and use directly.
4. **Missing infrastructure?** No new typeclass infrastructure is
   needed. If the worker discovers that `berlekampFactor` factors'
   per-element monicness is not yet exposed, that auxiliary lemma
   may be added in `HexBerlekamp/Factor.lean` as part of this
   issue (Mathlib-free, by induction on the loop) rather than
   filed as a separate dependency.

## Current state

`HexPolyMathlib/Basic.lean` defines the Mathlib bridge between
`Hex.DensePoly R` and `Polynomial R` via `toPolynomial` /
`ofPolynomial`. The `ofPolynomial_*` simp surface currently covers
`zero` (line 209), `one` (line 217), `C` (line 227), `neg` (line 239),
and `sub` (line 248); the `add` and `mul` cases are landing via #5253.
The matching `toPolynomial_*` cluster covers `zero`, `C`, `one`, `add`,
`neg`, `sub`, `mul`, and the round-trips `toPolynomial_ofPolynomial`
/ `ofPolynomial_toPolynomial`.

The monomial constructor is missing from both directions. Concretely:

- `Hex.DensePoly.monomial n c` is defined in `HexPoly/Dense.lean:138`,
  with the characterising lemma
  `Hex.DensePoly.coeff_monomial (n c i)` at `HexPoly/Dense.lean:188`:
  ```lean
  (monomial n c).coeff i = if i = n then c else (Zero.zero : R)
  ```
  and the simp lemma `monomial_zero` at `HexPoly/Dense.lean:374`.
- `Polynomial.monomial n c` is Mathlib's standard linear-map monomial;
  its characterising lemma is `Polynomial.coeff_monomial` (already used
  at `HexPolyMathlib/Basic.lean:199, 207`).

Because both representations share the same coefficient pattern
("`c` at index `n`, zero elsewhere"), `ofPolynomial (Polynomial.monomial n c)`
and `Hex.DensePoly.monomial n c` agree pointwise on every coefficient,
and the bridge should normalise the Mathlib-side monomial to the
executable-side monomial under simp.

## Deliverables

Add **one** new `@[simp]` theorem to `HexPolyMathlib/Basic.lean`,
inserted immediately after `ofPolynomial_sub` (line 254) and before
`toPolynomial_zero` (line 257) — or, if #5253 has already landed by
the time this issue is claimed, immediately after the new
`ofPolynomial_mul` declaration, in the same block:

```lean
/-- `ofPolynomial` sends Mathlib's monomial to the executable monomial. -/
@[simp]
theorem ofPolynomial_monomial [Semiring R] [DecidableEq R] (n : Nat) (c : R) :
    ofPolynomial (Polynomial.monomial n c) = Hex.DensePoly.monomial n c := by
  apply Hex.DensePoly.ext_coeff
  intro i
  rw [coeff_ofPolynomial, Hex.DensePoly.coeff_monomial,
      Polynomial.coeff_monomial]
  by_cases hi : i = n
  · simp [hi]
  · simp [hi]
```

The exact tactic shape can vary; the constraint is one `@[simp]`-tagged
theorem with that statement, proven without `sorry`, `axiom`, or
`native_decide`. Use the weakest typeclass that compiles; `Semiring R`
should suffice because `Polynomial.coeff_monomial` and
`Hex.DensePoly.coeff_monomial` both work in the semiring setting and
`ofPolynomial`'s definition requires only `Semiring R` + `DecidableEq R`.

Do not introduce a `toPolynomial_monomial` companion in this issue; that
is the symmetric follow-up and belongs in its own issue (a separate
worker can pick it up in parallel once this one is open).

Do not modify any other theorem, definition, import, or namespace in
`HexPolyMathlib/Basic.lean`. Insertion only.

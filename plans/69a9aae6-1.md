## Current state

`scripts/status.py` reports `HexHensel -> Phase 6 (proof polishing)`
ready. The closed Phase 6 review issue #5976 was a duplicate forward to
`reports/hex-hensel-lift-api-phase6-review.md`, which identified three
concrete polish follow-ups; none of those have been opened as issues.

This issue addresses recommendation #2 from that report:

> `henselLiftQuadratic_spec` only returns product congruence. Add a
> companion theorem for `(lifted.s * lifted.g + lifted.t * lifted.h) ≡
> 1 (mod p^k)` if downstream code needs the final witnesses directly.

Today `HexHensel/QuadraticMultifactor.lean` exposes only
`henselLiftQuadratic_spec` (`lifted.g * lifted.h ≡ f (mod p^k)`). The
loop already preserves the Bezout half through every doubling step via
`QuadraticLiftLoopInvariant.bezout_congr`, but there is no public
theorem that descends the Bezout witness from the loop modulus
`p ^ (2 ^ quadraticDoublingSteps k)` down to the requested precision
`p^k` and through the final `reduceModPow` on `lifted.s` and
`lifted.t`. Consumers that want to use the lifted Bezout pair at the
target precision (rather than at the loop modulus) currently have to
re-prove this descent themselves.

## Deliverables

In `HexHensel/QuadraticMultifactor.lean`, add a public theorem alongside
`henselLiftQuadratic_spec`:

```lean
theorem henselLiftQuadratic_bezout_spec
    (p k : Nat) [ZMod64.Bounds p]
    (f g h s t : ZPoly)
    (hk : 1 ≤ k)
    (hp : 1 < p)
    (hinv : QuadraticLiftLoopInvariant p f { g, h, s, t }) :
    let lifted := henselLiftQuadratic p k f g h s t
    ZPoly.congr (lifted.s * lifted.g + lifted.t * lifted.h) 1 (p ^ k)
```

Argument order, parameter names, and hypothesis bundling should match
`henselLiftQuadratic_spec` exactly so the two theorems read as a pair.
Add a docstring naming the inputs, the output, and the precondition the
caller must supply, in the same style as the existing
`henselLiftQuadratic_spec` docstring.

The proof should mirror the structure of `henselLiftQuadratic_spec`:

1. Run `iterateQuadraticHensel_invariant` to obtain the full
   `QuadraticLiftLoopInvariant` at modulus `p ^ (2 ^ fuel)`.
2. Project the Bezout half via
   `QuadraticLiftLoopInvariant.bezout_congr`.
3. Descend to modulus `p^k` via `congr_of_pow_le` and
   `le_two_pow_quadraticDoublingSteps`.
4. Handle the final `reduceModPow` on each of `s, g, t, h` using
   `congr_mul_reduceModPow_pair` (twice — once for `s * g`, once for
   `t * h`) and combine with `ZPoly.congr_add` to land the sum.

Do not change `henselLiftQuadratic`, `QuadraticLiftLoopInvariant`,
`iterateQuadraticHensel`, or any existing public theorem signatures.
Do not introduce a new helper file. Keep the new theorem private-free
(it is the consumer-facing companion to `henselLiftQuadratic_spec`).

Do not introduce `sorry`, `axiom`, `native_decide`, `TODO`, or `FIXME`.

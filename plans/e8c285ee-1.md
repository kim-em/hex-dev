## Current state

`HexHensel/Multifactor.lean` exposes the public `Array.polyProduct`
definition and three small algebra lemmas:

- `Hex.ZPoly.one_mul_zpoly` (line 93)
- `Hex.ZPoly.polyProduct_singleton` (line 98)
- `Hex.ZPoly.list_foldl_mul_eq_mul_foldl_one` (line 105)
- `Hex.ZPoly.polyProduct_singleton_append` (line 125)

The HexHensel surface is missing three further algebra lemmas that
downstream consumers actually need. `HexBerlekampZassenhaus/Basic.lean`
has therefore re-declared the same lemmas as `private` helpers at
lines 1919-1959, and extends them with three additional shapes:

| Private in BZ/Basic.lean             | Status vs. HexHensel/Multifactor.lean      |
|---|---|
| `one_mul_zpoly` (L1919)              | exact duplicate of the public lemma         |
| `list_foldl_mul_eq_mul_foldl_one` (L1923) | exact duplicate of the public lemma         |
| `polyProduct_singleton` (L1945)      | exact duplicate of the public lemma         |
| `polyProduct_cons_toArray` (L1940)   | new shape — `(g :: rest).toArray`            |
| `polyProduct_empty` (L1949)          | new shape — `Array.polyProduct #[] = 1`      |
| `polyProduct_append` (L1953)         | new shape — `polyProduct (xs ++ ys) = ...`   |

The three new shapes are not BZ-specific algebra; they are general
characterising lemmas for `Array.polyProduct` that belong in the same
file as `Array.polyProduct` itself. Consumers in
`HexBerlekampZassenhaus/Basic.lean` use them at 20+ call sites
(`polyProduct_contentFactorArray`, `polyProduct_repeatedPartFactorArray`,
`polyProduct_polynomialNormalizationPrefixFactors`, etc.); see lines
1966, 1974, 2100, 2113, 2127, 2165, 2171, 2313, 2317, 2327 for a
sample.

This is HexHensel Phase 6 work: `Array.polyProduct` is a public surface
and its core algebra lemmas should live next to it, not be re-derived
privately downstream.

## Deliverables

1. In `HexHensel/Multifactor.lean`, add three public characterising
   lemmas next to the existing `polyProduct_singleton` /
   `polyProduct_singleton_append`:

   ```lean
   theorem Hex.ZPoly.polyProduct_empty :
       Array.polyProduct (#[] : Array ZPoly) = 1

   theorem Hex.ZPoly.polyProduct_append (xs ys : Array ZPoly) :
       Array.polyProduct (xs ++ ys) =
         Array.polyProduct xs * Array.polyProduct ys

   theorem Hex.ZPoly.polyProduct_cons_toArray
       (g : ZPoly) (rest : List ZPoly) :
       Array.polyProduct (g :: rest).toArray =
         g * Array.polyProduct rest.toArray
   ```

   Port the proofs over from `HexBerlekampZassenhaus/Basic.lean`
   verbatim (they already use the public `one_mul_zpoly` and
   `list_foldl_mul_eq_mul_foldl_one`). Add short docstrings consistent
   with the existing `polyProduct_singleton{,_append}` docs.

   Decide and apply `[simp]` annotations where appropriate. `_empty`
   and `_singleton` look like natural `[simp]` candidates (they reduce
   a closed form to a normal-form scalar); `_append` and
   `_cons_toArray` are likely `[simp]` too. Check that adding
   `[simp]` does not break any existing proof by running the full
   downstream build (see Verification).

2. In `HexBerlekampZassenhaus/Basic.lean`, delete the now-redundant
   private duplicates:

   - `one_mul_zpoly` (L1919)
   - `list_foldl_mul_eq_mul_foldl_one` (L1923)
   - `polyProduct_cons_toArray` (L1940)
   - `polyProduct_singleton` (L1945)
   - `polyProduct_empty` (L1949)
   - `polyProduct_append` (L1953)

   The existing call sites resolve via the public names (they are in
   the `Hex.ZPoly` namespace via the surrounding `namespace Hex` /
   `namespace ZPoly` blocks); confirm there is no naming clash and
   that no further private-name qualification is needed. If a
   namespace clash forces it, the simplest fix is `open Hex.ZPoly in
   …` at the use site, not re-introducing the private declarations.

3. No new `axiom`, `native_decide`, `TODO`, `FIXME`, or theorem-level
   `sorry` introduced. Net line count change should be negative
   (about +25 in HexHensel/Multifactor.lean, about -40 in
   HexBerlekampZassenhaus/Basic.lean).

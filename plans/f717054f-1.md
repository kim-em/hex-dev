## Current state

`HexMatrix/RowEchelon.lean` defines column-add elementary column
operations via `Hex.Matrix.colAdd`. Phase 6 wrapper work has filled out
the row-side multiplication API:

- `rowAdd_mul` (`HexMatrix/RowEchelon.lean:366`) — proves
  `rowAdd A src dst s * B = rowAdd (A * B) src dst s` for
  `Lean.Grind.Ring R`.

The column-side mirror is currently missing. Downstream Phase 6 callers
that introduce column operations into a product on the right of a
matrix have to expand `colAdd` by hand. The natural mirror is:

```lean
theorem mul_colAdd
    (A : Matrix R n m) (B : Matrix R m k) (src dst : Fin k) (s : R) :
    A * colAdd B src dst s = colAdd (A * B) src dst s
```

The existing column readback wrappers (`col_colAdd_dst`,
`col_colAdd_of_ne`, `col_colAdd_src_of_ne`, `colAdd_getElem`,
`colAdd_getElem_src_of_ne`) and the row-side `rowAdd_mul` proof give a
direct template.

## Deliverables

1. In `HexMatrix/RowEchelon.lean`, add a private dot-product distributor
   over `Vector.ofFn` on the **right** factor, paralleling the existing
   `dotProduct_add_smul_ofFn_left` (around line 271):

   ```lean
   private theorem dotProduct_add_smul_ofFn_right [Lean.Grind.Ring R]
       (u v w : Vector R m) (s : R) :
       Hex.Vector.dotProduct u (Vector.ofFn fun k => v[k] + s * w[k]) =
       Hex.Vector.dotProduct u v + s * Hex.Vector.dotProduct u w
   ```

   Prove it with the same `foldl_sum_*_aux` lemmas (`foldl_sum_congr_aux`,
   `foldl_sum_mul_left_aux`, `foldl_sum_add_aux`) already used by the
   left version. Keep the private visibility — this is an internal
   distributivity helper, not part of the public surface.

2. Adjacent to `rowAdd_mul`, add the column-side wrapper:

   ```lean
   theorem mul_colAdd [Lean.Grind.Ring R]
       (A : Matrix R n m) (B : Matrix R m k) (src dst : Fin k) (s : R) :
       A * colAdd B src dst s = colAdd (A * B) src dst s
   ```

   The proof should follow `rowAdd_mul`'s structure: extensionality on
   entries, `mul_getElem` on both sides, `colAdd_getElem` on the right,
   case split on `j = dst`, and then use the readback lemmas
   (`col_colAdd_dst` for the `dst` branch, `col_colAdd_of_ne` for the
   other branch) together with `dotProduct_add_smul_ofFn_right` to
   close the `j = dst` case.

3. Keep the issue scoped to this one column-multiplication wrapper plus
   the necessary private helper. Do not add `colAdd_mulVec_getElem`,
   `colAdd_colAdd_neg`, `colAdd_left_inverse_preserve`,
   `colAdd_right_inverse_preserve`, `colAdd_transform_mul_preserve`,
   or `det_colAdd`-flavoured wrappers in this issue — those can be
   filed separately if needed.

## Library placement

Target file:

- `HexMatrix/RowEchelon.lean`

SPEC section: `SPEC/Libraries/hex-matrix.md` — the elementary
row/column operation API and its compatibility with matrix
multiplication.

Placement questions:

- Which SPEC section governs this file? The `HexMatrix` elementary
  column-operation API on `Hex.Matrix.colAdd`.
- Does the natural strategy use Mathlib? No. `HexMatrix` is
  Mathlib-free; the helper uses the existing project-local
  `Hex.Vector.dotProduct`, `Vector.ofFn`, and `Lean.Grind.Ring`
  laws already used by the row-side counterpart.
- Is this result already in Mathlib? Mathlib has analogous results
  for its own `Matrix` type, but this is the project-local wrapper
  for the executable `Hex.Matrix` representation.
- Does the deliverable presuppose missing infrastructure? No. The
  existing `foldl_sum_*_aux` helpers and column readback lemmas are
  already in the same file.

`HexPolyFp/Basic.lean` already defines the user-facing wrappers
(lines 856–862):

```lean
/-- Constant polynomial in `F_p[x]`. -/
def C (c : ZMod64 p) : FpPoly p :=
  DensePoly.C c

/-- The polynomial indeterminate `X`. -/
def X : FpPoly p :=
  DensePoly.monomial 1 (1 : ZMod64 p)
```

It also already has the evaluation lemmas `FpPoly.eval_C`, `FpPoly.eval_X`,
`FpPoly.eval_monomial` (lines 912–955), but the **coefficient** side of the
wrapper API is exposed only through `DensePoly.coeff_C` and
`DensePoly.coeff_monomial`. Callers reaching for `(FpPoly.C c).coeff n` or
`(FpPoly.X : FpPoly p).coeff n` currently have to either `unfold FpPoly.C` /
`unfold FpPoly.X` and reuse the generic dense-poly lemma, or write the
case-split by hand. The private helper `coeff_one` at line 907 confirms the
shape Lean uses, but it is not exported.

This was identified as the first follow-up slice in the merged Phase 6
review of `HexPolyFp` (issue #5131; review comment on the merged review PR
of issue #5131). The other three review slices —
`linearPow` helpers, quotient `% g` aliases, and a quotient/Frobenius
bridge — are deliberately not in scope here.

## Deliverables

1. In `HexPolyFp/Basic.lean`, immediately after the existing
   `eval_C`/`eval_X` cluster (~line 955), add public docstringed lemmas
   characterising the **coefficient sequence** of `FpPoly.C` and
   `FpPoly.X`. Concretely:
   - `theorem FpPoly.coeff_C (c : ZMod64 p) (n : Nat) :
       (FpPoly.C c).coeff n = if n = 0 then c else 0`
     proved by unfolding `FpPoly.C` and reusing `DensePoly.coeff_C`.
   - `theorem FpPoly.coeff_X_zero :
       ((FpPoly.X : FpPoly p)).coeff 0 = 0` and
     `theorem FpPoly.coeff_X_one :
       ((FpPoly.X : FpPoly p)).coeff 1 = 1`, proved by unfolding
     `FpPoly.X` and reusing `DensePoly.coeff_monomial` (matching the
     existing pattern at line 919).
   - `theorem FpPoly.coeff_X (n : Nat) :
       ((FpPoly.X : FpPoly p)).coeff n = if n = 1 then 1 else 0`
     stated in the same shape as `coeff_C`, again via
     `DensePoly.coeff_monomial`.

2. Mark **only the coefficient lemmas** with `@[simp]`. Do not add
   `@[simp]` to anything else. The review's simp-risk note explicitly
   restricts safe simp candidates here to coefficient lemmas with a
   small `if`-shaped RHS.

3. Each new theorem must have a one-line docstring on the line above
   `theorem`. Do not introduce new private helpers; reuse `DensePoly.coeff_C`
   and `DensePoly.coeff_monomial`. Do not unfold `DensePoly.C`/`monomial`
   beyond what those generic lemmas already require.

## Current state

`HexPoly/Dense.lean` exposes `DensePoly.support` together with the
two characterising simp lemmas:

- `@[simp] theorem support_zero : (0 : DensePoly R).support = []`
- `@[simp] theorem support_C (c : R) :
    (C c).support = if c = (0 : R) then [] else [0]`

and the membership lemma `mem_support_C` alongside it. The analogous
monomial-support surface is still indirect: callers must reason via
`mem_support`, `coeff_monomial`, and `size_monomial_of_ne_zero` to get
the support of `monomial n c`. This mirrors the recently added
constant-support pair (#5333) and is a natural Phase 6 characterising
extension.

## Deliverables

1. In `HexPoly/Dense.lean`, add a direct simp lemma for the support of
   a monomial, near the existing `support_zero` / `support_C` pair:

   ```lean
   @[simp] theorem support_monomial (n : Nat) (c : R) :
       (monomial n c).support = if c = (0 : R) then [] else [n] := ...
   ```

   Equivalent statement shapes are fine (e.g. branching by `c = 0` in
   the proof) but the `@[simp]` form must close with `simp` from the
   existing lemmas and produce the literal `[n]` / `[]` outputs without
   relying on the caller to peel a `Decidable` instance.

2. Add the membership counterpart, mirroring `mem_support_C`:

   ```lean
   theorem mem_support_monomial {n : Nat} {c : R} {i : Nat} :
       i ∈ (monomial n c).support ↔ i = n ∧ c ≠ (0 : R) := ...
   ```

   `mem_support_C` is unannotated — match that. Do not mark this one
   `@[simp]`.

3. Keep the change local to `HexPoly/Dense.lean`; do not modify
   operations, Euclidean division, conformance fixtures, or benchmarks.
   No other simp lemmas should be added in this issue — broader support
   coverage (e.g. `support_add`, `support_mul`) is out of scope.

## Library placement

Target file: `HexPoly/Dense.lean`.

SPEC section: `SPEC/Libraries/hex-poly.md`:
"Dense representation" and the "The normalization invariant (no
trailing zeros) ensures structural equality = semantic equality.
Every operation maintains this invariant."

Placement questions:

- Which SPEC section governs this file? `SPEC/Libraries/hex-poly.md`,
  dense representation and basic polynomial observation API.
- Does the natural strategy use Mathlib? No. This is Mathlib-free API
  polish over the existing executable `monomial` definition.
- Is this result already in Mathlib? No. `DensePoly` is a project-local
  type; the analogous Mathlib lemma is `Polynomial.support_monomial`,
  but we are characterising the Hex executable surface.
- Does the deliverable presuppose missing infrastructure? No.
  `support`, `monomial`, `coeff_monomial`, `mem_support`, and
  `size_monomial_of_ne_zero` are already present.

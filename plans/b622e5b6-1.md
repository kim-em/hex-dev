## Current state

`HexBerlekampZassenhausMathlib/Basic.lean` L3142 contains
`henselLiftData_liftedFactor_monic`, the umbrella wrapper landed via
PR #4390. Its signature requires the abstract loop invariant:

```lean
theorem henselLiftData_liftedFactor_monic
    (core : Hex.ZPoly) (B : Nat) (primeData : Hex.PrimeChoiceData)
    (hcore_monic : Hex.DensePoly.Monic core)
    (hprime_invariant :
      letI := primeData.bounds
      Hex.ZPoly.QuadraticMultifactorLiftInvariant
        primeData.p B core
        (primeData.factorsModP.map Hex.FpPoly.liftToZ).toList)
    (hp : 1 < primeData.p)
    (hB : 1 ≤ B) :
    ∀ i : Fin (Hex.henselLiftData core B primeData).liftedFactors.size,
      Hex.DensePoly.Monic
        (Hex.henselLiftData core B primeData).liftedFactors[i]
```

`Hex.ZPoly.QuadraticMultifactorLiftInvariant` is the wrong API for a
Mathlib-bridge caller: it is the internal loop invariant of
`HexHensel/QuadraticMultifactor.lean`, not something a public coverage
caller has on hand. The natural shape a caller can supply is the
`choosePrimeData` boundary facts: factors-mod-p are monic, their lifted
product is congruent to `core` mod `p`, and pairwise coprimality.

Issue #4393 is currently claimed and produces, in
`HexHensel/QuadraticMultifactor.lean`, the bridge
`Hex.ZPoly.QuadraticMultifactorLiftInvariant_of_choosePrimeData`
(name negotiable) from those boundary facts. Once that lands, a
Mathlib-bridge consumer still has to compose it with the existing
umbrella to obtain `∀ i, Monic (liftedFactor d i)`. Every public
coverage caller (the eventual #4274 entry point and any direct
HenselSubsetCorrespondence consumer) would have to repeat that
composition.

This issue lands the composed convenience wrapper at the Mathlib-bridge
layer so the boundary-facts-to-liftedFactor-monic step is a single
named theorem call.

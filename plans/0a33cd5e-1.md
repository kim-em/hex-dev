`HexGfqRing/Operations.lean` is 1094 lines and naturally decomposes
into five sub-surfaces. This review covers exactly one of them: the
**`ofPoly`/`reduceMod` simp normal form cluster** that ships the
"operations commute with `ofPoly` / `reduceMod` / `repr`" lemmas a
downstream caller uses to push reduction past `+ * - neg` and pull
`ofPoly` out of operator expressions.

Concretely the cluster spans roughly lines 314–399 and includes:

- `reduceMod_add_reduceMod` and `reduceMod_mul_reduceMod` — public
  aliases for the underlying congruence lemmas
- `reduceMod_repr` (`@[simp]`) — canonical representative is already
  reduced
- `ofPoly_zero_eq_zero`, `ofPoly_one_eq_one`, `ofPoly_const_eq_const`
  (`@[simp]`) — `ofPoly` interactions with constants
- `ofPoly_add_reduceMod`, `ofPoly_mul_reduceMod`,
  `ofPoly_neg_reduceMod`, `ofPoly_sub_reduceMod` — non-`@[simp]`
  rewriting forms in the `ofPoly` direction
- `repr_add_ofPoly`, `repr_mul_ofPoly`, `repr_neg_ofPoly`,
  `repr_sub_ofPoly` (`@[simp]`) — pull `repr (ofPoly … ⋆ ofPoly …)`
  down to `reduceMod` on the raw coefficients

The `@[simp]` story today is asymmetric — `repr_*_ofPoly` are simp,
the `ofPoly_*_reduceMod` "opposite direction" lemmas are not, the
two `reduceMod_*_reduceMod` aliases are not. That asymmetry may be
intentional (to converge a simp set toward `repr (… : PolyQuotient)`
as the normal form on the right-hand side), but it has not been
audited.

SPEC §: `SPEC/Libraries/hex-gfq-ring.md` "Key properties" pins this
file's responsibilities:

> - `reduceMod f (a + b) = reduceMod f (reduceMod f a + reduceMod f b)`
> - `reduceMod f (a * b) = reduceMod f (reduceMod f a * reduceMod f b)`
> - Ring axioms for `PolyQuotient p f hf`

Placement questions:

1. **SPEC §:** the lines quoted above name the cluster directly;
   this is the file the SPEC assigns the cluster to.
2. **Mathlib?** No. HexGfqRing is a Mathlib-free library and these
   lemmas are about the `FpPoly p` representative.
3. **Already in Mathlib?** No — this surface is hex-specific
   plumbing for the canonical-representative `PolyQuotient` type;
   Mathlib's quotient-ring API uses `Quot`/`Setoid`, not a
   canonical-representative subtype.
4. **Missing infrastructure?** No — every declaration named here
   already exists; this is a polish review, not an addition.

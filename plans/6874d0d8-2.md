## Current state

`HexPoly/Operations.lean` exposes Horner-style `DensePoly.eval`
together with the simp wrappers:

- `@[simp] theorem eval_zero` — `eval (0 : DensePoly R) x = 0`
- `@[simp] theorem eval_C_semiring (c x : S) : eval (C c) x = c`

The `eval` surface is otherwise missing direct simp characterisations
for the other elementary constructors. In particular, the natural
companion `eval_monomial_semiring` — covering `eval (monomial n c) x`
— is absent, so callers must unfold `monomial` or hand-thread Horner
steps. The pattern mirrors the recent zero/`C` Phase 6 additions
(#5333, #5334) and the generic/`_semiring`-pair convention already
used by `coeff_scale_semiring`, `coeff_add_semiring`, and
`coeff_derivative_semiring`.

## Deliverables

1. In `HexPoly/Operations.lean`, add a semiring-specialized simp
   wrapper for evaluating a monomial:

   ```lean
   @[simp] theorem eval_monomial_semiring {S : Type u}
       [Lean.Grind.Semiring S] [DecidableEq S]
       (n : Nat) (c x : S) :
       eval (monomial n c) x = c * x ^ n := ...
   ```

   Statement shape may be adjusted (e.g. `x ^ n * c`) only if necessary
   to match `Lean.Grind.Semiring`'s normal form; prefer `c * x ^ n` to
   align with the `coeff_monomial` characterisation `if i = n then c
   else 0` and Mathlib's `Polynomial.eval_monomial`. Branch on
   `c = 0` internally if needed (the `c = 0` case collapses via
   `monomial_zero` and `eval_zero`).

2. If the proof naturally exposes a single-line generic helper
   (analogous to the existing `eval_C` paired with `eval_C_semiring`),
   it is fine to introduce one unannotated `theorem eval_monomial`
   parameterized over the zero laws. Do **not** add multiple generic
   variants or simp lemmas for `Add R` / `Mul R` without
   `Lean.Grind.Semiring` — match the existing `eval_C` / `eval_C_semiring`
   split exactly.

3. Keep the change local to `HexPoly/Operations.lean`; do not modify
   `HexPoly/Dense.lean`, Euclidean division, conformance fixtures, or
   benchmarks. Do not add `eval_add_semiring`, `eval_scale_semiring`,
   `eval_X`, or other `eval` simp lemmas in this issue — they are
   separate Phase 6 items.

## Library placement

Target file: `HexPoly/Operations.lean`.

SPEC section: `SPEC/Libraries/hex-poly.md`:
"Operations: ... Evaluation (Horner's method) ... Composition,
derivative".

Placement questions:

- Which SPEC section governs this file? `SPEC/Libraries/hex-poly.md`,
  Operations subsection naming evaluation as a core operation.
- Does the natural strategy use Mathlib? No. The lemma is purely about
  the executable `eval` / `monomial` definitions; the analogous
  Mathlib statement is `Polynomial.eval_monomial`, but we are
  characterising the Hex executable surface.
- Is this result already in Mathlib? Yes, in the analogous form
  `Polynomial.eval_monomial`. Strategy: reprove locally on the Hex
  type. The Hex `eval` is Horner-foldl over the normalised coefficient
  array, distinct from Mathlib's `Polynomial.eval`, so direct import
  is not available — reproving here is the right path, not a
  blank-page redo.
- Does the deliverable presuppose missing infrastructure? No.
  `eval`, `monomial`, `coeff_monomial`, `eval_C` /
  `eval_C_semiring`, `monomial_zero`, `eval_zero`, and
  `coeffs_C_of_ne_zero` are already present.

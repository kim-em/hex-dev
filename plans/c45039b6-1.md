## Current state

PR #4390 (closes #4376) adds the umbrella wrapper
`HexBerlekampZassenhausMathlib.henselLiftData_liftedFactor_monic`.
The wrapper takes `Monic core` plus
`Hex.ZPoly.QuadraticMultifactorLiftInvariant primeData.p B core
(primeData.factorsModP.map Hex.FpPoly.liftToZ).toList` and concludes
that every `(Hex.henselLiftData core B primeData).liftedFactors[i]` is
monic, by delegating to `Hex.ZPoly.multifactorLiftQuadratic_each_monic`
(merged via PR #4386).

`#4376` deliverable 2 explicitly carved out, as optional follow-up:

> Add a companion theorem that derives
> `QuadraticMultifactorLiftInvariant` from the more natural inputs that
> arise at the `choosePrimeData` boundary: `∀ g ∈ primeData.factorsModP,
> FpPoly.Monic g` plus the product-mod-p identity. This is what callers
> will actually have. Defer to a follow-up sub-issue if it requires
> substantial surgery in `HexHensel/QuadraticMultifactor.lean`.

The PR #4390 author confirmed in the session progress note that this
companion theorem was deferred; the umbrella wrapper as landed is only
useful to callers that already hold the abstract loop invariant.

The downstream consumer chain — #4377 (integer-factor monic capstone,
blocked on #4376), then #4301 (recursive rest-search coverage), then
#4274 (public coverage theorem) — eventually has to discharge
`∀ i, Hex.DensePoly.Monic (liftedFactor d i)` from inputs that arise
at the `Hex.choosePrimeData` boundary inside
`HenselSubsetCorrespondenceHypotheses`. Without this companion, every
public coverage statement has to thread `QuadraticMultifactorLiftInvariant`
itself through, which is the wrong API.

This issue lands the companion bridge so the umbrella from #4376 is
actually invocable from the public coverage chain.

## Deliverables

1. In `HexHensel/QuadraticMultifactor.lean` (or a new neighbour file
   under `HexHensel/` if the proof is large enough to deserve its own
   module), prove a companion theorem with shape

   ```lean
   theorem Hex.ZPoly.quadraticMultifactorLiftInvariant_of_factorsModP
       (p k : Nat) [ZMod64.Bounds p]
       (f : Hex.ZPoly) (factors : List (Hex.FpPoly p))
       (hp : 1 < p)
       (hk : 1 ≤ k)
       (hf_monic : Hex.DensePoly.Monic f)
       (hfactors_monic : ∀ g ∈ factors, Hex.FpPoly.Monic g)
       (hproduct_mod_p :
         Hex.ZPoly.congr
           (Array.polyProduct
             ((factors.map Hex.FpPoly.liftToZ).toArray))
           f p)
       (hpairwise_coprime :
         /- the mod-p coprimality side condition required by
            `normalizedXGCD` along the recursive split — exact form to
            be pinned down by inspection of `multifactorLiftQuadraticList`
            and `normalizedXGCD` -/) :
       Hex.ZPoly.QuadraticMultifactorLiftInvariant p k f
         (factors.map Hex.FpPoly.liftToZ) := by
     sorry
   ```

   Adjust the hypothesis list to whatever shape is cleanest. The
   intended caller has `Monic core` and the `berlekampFactorsModP`
   guarantees from `Hex.choosePrimeData` (factors monic mod p, product
   mod p equals `core mod p`, pairwise coprime mod p / squarefreeness
   inherited from `isGoodPrime`).

2. Derive a `Hex.ZPoly.QuadraticMultifactorLiftInvariant_of_choosePrimeData`
   convenience wrapper (name negotiable) that specialises step 1 to the
   `(primeData : Hex.PrimeChoiceData)` case used by
   `HexBerlekampZassenhausMathlib.henselLiftData_liftedFactor_monic`,
   so a Mathlib-side caller can write

   ```lean
   have hinv :
       Hex.ZPoly.QuadraticMultifactorLiftInvariant primeData.p B core
         (primeData.factorsModP.map Hex.FpPoly.liftToZ).toList :=
     Hex.ZPoly.QuadraticMultifactorLiftInvariant_of_choosePrimeData
       core B primeData hcore_monic hpData hp hB
   ```

   for a single packaged hypothesis `hpData` capturing the
   `choosePrimeData` boundary facts. If those facts are already
   captured by an existing executable theorem about `choosePrimeData`,
   reuse it; otherwise expose the minimal new private helper inside
   `HexBerlekampZassenhaus/Basic.lean` and re-export only the public
   convenience wrapper.

3. The proof is by induction on `factors`:
   - `[]` case: the invariant unfolds to `ZPoly.congr 1 f (p ^ k)`,
     which follows from `hproduct_mod_p` (the product over `[]` is `1`)
     and `Hex.ZPoly.congr_pow_of_congr` style monotonicity, given that
     `hf_monic` forces a degree-zero `f`. Pin the exact lemma when
     proof develops.
   - `[_g]` case: the invariant unfolds to `True`.
   - `g :: rest` (with `rest ≠ []`) case: unfold the invariant to a
     conjunction of `QuadraticLiftLoopInvariant p f {g, h, s, t}` and
     `QuadraticMultifactorLiftInvariant p k lifted.h rest`. Discharge
     the loop invariant from the per-factor monicness, the Bezout
     identity from `normalizedXGCD`'s spec on the coprime side
     condition, and the product congruence from `hproduct_mod_p`.
     Recurse on the lifted complementary factor using
     `henselLiftQuadratic_spec`-style facts already available in
     `HexHensel/QuadraticMultifactor.lean`.

4. Add a unit-style `example` (or short consumer theorem) at the end
   of `HexBerlekampZassenhausMathlib/Basic.lean` that combines the
   convenience wrapper from step 2 with `henselLiftData_liftedFactor_monic`
   to derive `∀ i, Hex.DensePoly.Monic (liftedFactor d i)` from
   `Monic core` plus the `choosePrimeData`-boundary hypothesis package
   alone. This is the actual signature that #4377 / #4301 / #4274 will
   want to consume; landing it here (even if as a thin abbreviation)
   prevents every downstream theorem from re-doing the chain.

5. No new `axiom`, `native_decide`, `TODO`, `FIXME`, or theorem-level
   `sorry` in the diff.

## Library placement

- `Hex.ZPoly.quadraticMultifactorLiftInvariant_of_factorsModP` and the
  recursive proof body belong in `HexHensel/QuadraticMultifactor.lean`
  alongside `QuadraticMultifactorLiftInvariant` and
  `multifactorLiftQuadratic_each_monic`. This file is Mathlib-free.
- The `_of_choosePrimeData` convenience wrapper depends on
  `Hex.PrimeChoiceData` (which lives in `HexBerlekampZassenhaus/Basic.lean`),
  so it goes there. Still Mathlib-free.
- The end-to-end consumer combinator in step 4 belongs in
  `HexBerlekampZassenhausMathlib/Basic.lean`, immediately after
  `henselLiftData_liftedFactor_monic`.

The `Mathlib-free / Mathlib-bridge` boundary is preserved: the proof
chain through `HexHensel/QuadraticMultifactor.lean` and
`HexBerlekampZassenhaus/Basic.lean` does not import Mathlib; the
Mathlib-side consumer in `HexBerlekampZassenhausMathlib/Basic.lean` is
where the lemma combines with the existing `Monic` predicate that the
downstream coverage chain consumes.

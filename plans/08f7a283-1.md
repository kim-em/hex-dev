## Current state

`HexPolyFp/Basic.lean` defines `instDivModLawsZMod64Fp`, the
`DensePoly.DivModLaws (ZMod64 p)` instance used by every downstream
finite-field quotient construction. Most fields are discharged via
private `_core` helpers, but four `sorry`s remain — two in the monic /
trivial-divisor cluster and two in the gcd cluster:

```
  divModMonic_eq_divMod_of_monic     -- HexPolyFp/Basic.lean:606
  mod_self_eq_zero                   -- HexPolyFp/Basic.lean:609
  mod_eq_zero_of_dvd                 -- HexPolyFp/Basic.lean:612
  mod_mod_of_not_pos_degree          -- HexPolyFp/Basic.lean:615
```

This issue covers the two fields whose `_core` lemmas already exist in
`HexPoly/Euclid.lean`:

- `divModMonic_eq_divMod_of_monic` reduces to
  `DensePoly.divModMonic_eq_divMod_of_monic_core` once we know
  `∀ a : ZMod64 p, a / q.leadingCoeff = a` for monic `q` (i.e.
  `a / 1 = a` in `ZMod64 p`).
- `mod_mod_of_not_pos_degree` reduces to either
  `DensePoly.divMod_remainder_eq_self_of_size_zero_core` (when
  `q.size = 0`) or
  `DensePoly.divMod_remainder_eq_zero_of_degree_zero_core` (when
  `q.size = 1`), the latter requiring the `hcancel` witness
  `∀ a : ZMod64 p, a - (a / q.leadingCoeff) * q.leadingCoeff = 0` which
  the existing `divMod_remainder_degree_lt_core` proof in this file
  already establishes (lines 142–155) for nonzero `q.leadingCoeff` over
  a `PrimeModulus`.

The other two `sorry`s (`mod_self_eq_zero`, `mod_eq_zero_of_dvd`) and the
`GcdLaws` cluster (lines 664/667/670/673) are out of scope here; they
need separate issues with their own helper-lemma plans.

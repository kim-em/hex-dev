## Current state

`HexPolyMathlib/Basic.lean` exposes the executableŌåöMathlib polynomial bridge,
with the `toPolynomial_*` simp wrappers for every elementary operation already
in place ŌĆö `toPolynomial_zero`, `toPolynomial_C`, `toPolynomial_one`,
`toPolynomial_add`, `toPolynomial_neg`, `toPolynomial_sub`, `toPolynomial_mul`
(`HexPolyMathlib/Basic.lean:227-294`). The previous Phase 6 batch (PR #5171,
issue #5166) closed `toPolynomial_one`, `toPolynomial_neg`, `toPolynomial_sub`,
and `ofPolynomial_one`.

The `ofPolynomial_*` side is now lagging: only `ofPolynomial_zero` and
`ofPolynomial_one` are present (lines 210, 218). The constructor / negation /
subtraction wrappers that mirror the `toPolynomial_*` side are missing, so
proofs that thread `ofPolynomial` over `C`, `-`, or `-` cannot rely on simp to
push the bridge through and must call `Hex.DensePoly.ext_coeff` by hand.

## Deliverables

Extend `HexPolyMathlib/Basic.lean` with **three** new `@[simp]` theorems, placed
in the existing `ofPolynomial_*` cluster (immediately after
`ofPolynomial_one`, lines 217-224) to keep the cluster contiguous. The
typeclass surface should match the corresponding `toPolynomial_*` lemma:

```lean
/-- `ofPolynomial` sends Mathlib's polynomial constant to the executable
constant. -/
@[simp]
theorem ofPolynomial_C [Semiring R] [DecidableEq R] (c : R) :
    ofPolynomial (Polynomial.C c) = Hex.DensePoly.C c := ...

/-- `ofPolynomial` commutes with polynomial negation. -/
@[simp]
theorem ofPolynomial_neg [Ring R] [DecidableEq R] (p : Polynomial R) :
    ofPolynomial (-p) = -ofPolynomial p := ...

/-- `ofPolynomial` commutes with polynomial subtraction. -/
@[simp]
theorem ofPolynomial_sub [Ring R] [DecidableEq R] (p q : Polynomial R) :
    ofPolynomial (p - q) = ofPolynomial p - ofPolynomial q := ...
```

Suggested proof shape, mirroring `toPolynomial_neg` / `toPolynomial_sub`
(`HexPolyMathlib/Basic.lean:259-274`):

```lean
apply Hex.DensePoly.ext_coeff
intro n
rw [Hex.DensePoly.coeff_neg _ _ (by show (0 : R) - 0 = 0; simp)]
simp [coeff_ofPolynomial, Polynomial.coeff_neg]
```

The `ofPolynomial_C` proof should follow the `toPolynomial_C` shape
(`HexPolyMathlib/Basic.lean:233-240`): `Hex.DensePoly.ext_coeff` +
`Hex.DensePoly.coeff_C` + `Polynomial.coeff_C` + case split on `n = 0`.

Do **not** add `ofPolynomial_add` or `ofPolynomial_mul` in this issue ŌĆö
those are larger (the `add`/`mul` discharge proofs for `Hex.DensePoly.coeff_*`
are bulkier and the `mul` side requires the same antidiagonal sum manipulation
as `toPolynomial_mul`). They are explicit follow-up scope, not part of this
deliverable.

No new definitions, instances, imports, or file moves. Do not introduce
`sorry`, `axiom`, `native_decide`, `TODO`, or `FIXME`. Do not edit `SPEC/`,
top-level `PLAN.md`, or top-level `AGENTS.md`.

The `coeff_neg` / `coeff_sub` API takes an explicit discharge of the
`(0 : R) - 0 = 0` boundary condition; the existing `toPolynomial_neg` /
`toPolynomial_sub` proofs discharge it with
`by show (0 : R) - 0 = 0; simp`. The same discharge should work here. The
ZMod64-specific typeclass-shaped wrappers (`coeff_neg_ring`, `coeff_sub_ring`)
do **not** fire over arbitrary `Ring R`, so use the explicit-discharge form,
not the typeclass form.

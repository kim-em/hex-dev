Since PR #5169 (merged), `HexPoly/Operations.lean` exposes three
typeclass-specialized `@[simp]` wrappers that close the discharge
internally:

- `DensePoly.coeff_add_semiring`
  — `[Lean.Grind.Semiring S] [DecidableEq S] [AddZeroLaw S]`,
  `(p + q).coeff n = p.coeff n + q.coeff n`.
- `DensePoly.coeff_sub_ring`
  — `[Lean.Grind.Ring S] [DecidableEq S] [SubZeroLaw S]`,
  `(p - q).coeff n = p.coeff n - q.coeff n`.
- `DensePoly.coeff_neg_ring`
  — `[Lean.Grind.Ring S] [DecidableEq S] [SubZeroLaw S]
       [ZeroSubNegLaw S]`, `(-p).coeff n = -(p.coeff n)`.

`ZMod64 p` carries `Lean.Grind.Semiring` and `Lean.Grind.Ring`
instances (`HexModArith/Ring.lean:278, 368`); `HexPolyFp/Basic.lean:52,
59` defines the matching `AddZeroLaw (ZMod64 p)` and `SubZeroLaw
(ZMod64 p)` instances; `ZMod64 p` is `DecidableEq` via its single
`UInt64` field. `HexPoly.Operations` is already transitively imported
through `HexPolyFp.Basic` (which `HexBerlekamp/RabinSoundness.lean`
imports).

This is the call-site migration sibling of #5180 (`Compose.lean`),
#5181 (`Quotient.lean` / `QuotientFrobenius.lean`), #5182
(`Basic.lean`), #5183 (`SquareFree.lean`) — all four under
`HexPolyFp/`. The `HexBerlekamp/RabinSoundness.lean` migration
matches the same mechanical pattern at a single file but at larger
volume (34 sites vs ≤ 6 per sibling issue).

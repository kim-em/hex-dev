## Current frontier

- The main design decision is whether `HexResultant` should carry a second
  proof-oriented Sylvester implementation to obtain a cheap bridge theorem, or
  whether the project wants to invest in a heavier subresultant formalisation that
  is likely upstreamable.
- On the irreducibility side, the current SPEC already favours a checker/certificate
  story in `hex-berlekamp-zassenhaus` plus a Mathlib bridge for `Irreducible`, so a
  new Mathlib-free `Decidable (Hex.ZPoly.Irreducible f)` needs an explicit design
  decision rather than being inferred from the existing plan.

## Next step

- Convert this review into a concrete SPEC decision: either
  1. adopt dual implementation for resultants with a narrow bridge theorem, or
  2. explicitly budget a Mathlib-grade subresultant formalisation.
- If the project wants Mathlib-free irreducibility as an API, first tighten the

## Blockers

- The exact normalization contract of `HexBerlekampZassenhaus.factor` is not yet a
  strong enough theorem surface to support a clean `isIrreducible_iff` proof without
  further spec work.
- Any direct subresultant bridge depends on theorem infrastructure that neither the
  repo nor current Mathlib provides today.

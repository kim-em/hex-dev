## Accomplished

Claimed issue #3296 (HexArith Phase 6: review extended-GCD API) and made two
focused polishing changes to `HexArith/ExtGcd.lean`:

- Removed dead-code helpers `Hex.uint64ExtGcdLoop` (private) and
  `Hex.pureUInt64ExtGcd` (public). Neither was referenced outside the file,
  neither had characterising `_fst`/`_bezout` theorems, and the SPEC mandates
  that `HexArith.UInt64.extGcd` route through `HexArith.Int.extGcd` (it
  does), so `pureUInt64ExtGcd` is not the GMP-fallback path. Their
  presence violated the Phase 6 exit criterion "no dead or unreferenced
  declarations".
- Rephrased `HexArith.Int.extGcd_zero_left_s_ofNat` from the
  `match`-binding form
  `(match extGcd 0 (Int.ofNat p) with | (_, s, _) => s) = 0` to the
  direct projection form `(extGcd 0 (Int.ofNat p)).2.1 = 0`. The only
  caller (`HexBerlekampZassenhaus.Basic.zmod64_inv_zero`) previously needed
  a `change ... at hs` line to coerce between the two; that line is now
  unnecessary and was deleted.

The `extGcd_fst` / `extGcd_bezout` API surface (Nat / Int / UInt64) was
checked against SPEC `Libraries/hex-arith.md` and the downstream callers
in `HexModArith.Basic` (`inv_mul_eq_one`), `HexBerlekampZassenhaus.Basic`,
`HexArith.Conformance`, and `HexArith.Bench`. No further polish needed
in this pass — the projection/let shapes match what callers consume, and
adding speculative `@[simp]`/`@[grind]` attributes was not justified by
any concrete downstream proof pattern.

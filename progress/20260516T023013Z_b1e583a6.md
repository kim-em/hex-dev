## Accomplished

Claimed #4363 ("HexBZ Mathlib: Hensel-lifted factors are monic at
henselLiftData precision") via `/work` and oriented on the
infrastructure. Identified that the issue's "step 1" (`∀ i, Monic
(liftedFactor d i)` for `d = henselLiftData …`) is **not** a thin
wrapper around `QuadraticLiftLoopInvariant`, contrary to the issue
body's optimistic framing. The loop invariant
(`HexHensel/QuadraticMultifactor.lean` L83) carries `Monic acc.g`
only, so only the per-step "kept" factor inherits monicness. The
*final* output of `multifactorLiftQuadratic` is
`reduceModPow lifted_{n-1}.h p k` — the deepest cofactor — whose
monicness is not guaranteed by any existing invariant.

Surveyed the executable Hensel step
(`HexHensel/Quadratic.lean` L1780, `quadraticHenselStep`) and
confirmed the loop's `hCorrection` can have nonzero leading-degree
coefficients (multiples of `m`), so `h.coeff (deg h)` is *not*
mechanically preserved at `1` during the lift. Empirically the
algorithm still produces monic cofactors, because the unique Hensel
lift of monic factors of a monic polynomial is monic.

Identified the correct proof strategy: don't extend the loop
invariant, instead derive cofactor monicness *from the spec
contract* `congr (g * h) f m` plus `Monic f`, `Monic g`, and the
coefficient bound `0 ≤ h.coeff i < m` that
`reduceModPow` already guarantees on the output. The argument is a
degree-comparison + congruence-at-`deg f`:

1. If `h.size > n - d + 1` (where `n = deg f`, `d = deg g`), then
   the congruence at index `j = h.size - 1 + d > n` forces
   `h.coeff (h.size - 1) ≡ 0 (mod m)`, which combined with
   `0 ≤ h.coeff < m` and `h ≠ 0` (so the leading coefficient is
   nonzero) is a contradiction.
2. Then `h.size = n - d + 1` and the congruence at `j = n` forces
   `h.coeff (n - d) = 1`. This is exactly the leading coefficient.

Decomposed #4363 into a 4-stage pipeline rather than land a partial
proof:

  degree-comparison + congruence argument above).
- **#4374**: lift the lemma through `henselLiftQuadratic` and
  `multifactorLiftQuadratic` by induction. Depends on #4373.
- **#4376**: thin `henselLiftData` wrapper. Depends on #4374. This
  is step 1 of #4363.
- **#4377**: integer-factor monic capstone via
  `RepresentsIntegerFactorAtLift` + Mignotte precision. Depends on
  #4376. This is step 2 of #4363.

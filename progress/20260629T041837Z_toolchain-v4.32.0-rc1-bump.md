# Toolchain bump to v4.32.0-rc1

Branch: `bump-toolchain-v4.32.0-rc1`.

## Accomplished

- `lean-toolchain` → `leanprover/lean4:v4.32.0-rc1`; `verso` pin → `v4.32.0-rc1`.
- **Full source migration to v4.32.** ~120 fix sites across the tree, almost all
  the Lean 4.31 `simpa` change: the final goal/term match in `simpa using e` now
  uses *reducible* transparency, so it no longer unfolds plain `def`s. Pure
  defeq-bridge `simpa [...] using e` calls became `exact e`; where real rewriting
  plus a defeq residual was involved, the rewrite was split out so the closing
  step is `exact` at default transparency (`simp only [...] at h; exact h`, typed
  `have`, `rw`). A few genuine v4.32 API changes: `smul_apply` ambiguity
  (`Matrix.smul_apply`), `getElem!_def` now a `match` (local `getElem!_eq_getD`
  bridge), `congrArg` beta-reduces (drop `dsimp`/typed `have`), `simp` leaving
  `if True`/`if c` unreduced, and `Std.Time` moves (in `lean-bench`, below).
- **No new `sorry`/`axiom`** (diff vs `origin/main` is clean; the handful of
  pre-existing `sorry`s in `HexBerlekampZassenhausMathlib` / `HexBareissMathlib`
  are untouched and unrelated).

## The Lake "build cycle" was a batteries packaging bug, not a Hex/v4.32 issue

`batteries/Batteries:shared` self-cycles (`Batteries:shared ↔
BatteriesRecycling:shared`) at the batteries commit mathlib v4.32.0-rc1 pins
(`954dbc98`), because that lakefile declares `Batteries` and `BatteriesRecycling`
as separate, mutually-importing libs. Our `precompileModules` libraries request
that shared lib, surfacing it as a long Hex cycle. Dropping `precompileModules`
is NOT viable (the FFI `#guard`/cross-check modules need native code at
elaboration). Fixed upstream by **batteries#1868** ("move recycling to avoid
circular dependency", same v4.32.0-rc1 toolchain). Current **mathlib master is
still on the v4.32.0-rc1 toolchain and pins fixed batteries (`77d3cc51`)** with a
working cache, so the project builds green against mathlib master.

# 0032da05 — HexArith Barrett vs Montgomery cross-check

## Accomplished

- Closed issue #2006 by adding `HexArith/CrossCheck.lean`, a Tier-G
  fast-vs-fast cross-check between `BarrettCtx.mulMod` and
  `MontCtx.mulMont`.
- Streams (deterministic LCG, no IO randomness):
  - `mulStream` — 256 (a, b) pairs; first 8 also checked against
    plain `Nat` reference; all entries check Barrett ≡ Montgomery.
  - `powStream` — 100 (base, exp) pairs with exp ∈ [0, 1000]; checks
    Barrett-loop powMod ≡ Montgomery-loop powMod.
  - `invStream` — 100 nonzero residues; checks Barrett-extGcd inverse
    ≡ Montgomery `a^(p-2)` inverse.
- Verified divergence detection: temporarily injecting `+ 1` into the
  Barrett side of `mulModAgree` causes the `#guard` to fail with
  `"did not evaluate to 'true'"`; restored after verification.
- Wired the new module into `HexArith.lean` so the existing build
  pipeline picks it up.

## Decisions

- **Modulus choice.** The issue named `p ≈ 2^60` (e.g. `2^60 - 93`),
  but `BarrettCtx` requires `p < 2^32` per
  `SPEC/Libraries/hex-arith.md:26` and the structure's `p_lt` field.
  The shared regime for the two fast paths is therefore
  `p < 2^32 ∧ p % 2 = 1`. I used `p = 2^32 - 5 = 4294967291`, the
  largest prime below `2^32`, so every nonzero residue is invertible
  and Fermat's little theorem applies for `inverseStream`. Documented
  in the file's module docstring.
- **Stream size.** Dropped from 1000 to 256 entries per the issue's
  fallback instruction. Without `native_decide` (banned project-wide
  per SPEC), kernel evaluation does not fit 1000 cross-checks under
  the named 2 s elaboration target on the project's CI hardware.
  Comment in the file documents the deviation.

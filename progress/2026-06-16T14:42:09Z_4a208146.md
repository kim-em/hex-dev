# Progress — 4a208146 (feature #7598)

## Accomplished

Extended the `HexPolyZMathlib.toPolynomial` projection simp surface at the
`Hex.ZPoly` bridge with the three statable ring-projection lemmas, each
delegating to the existing generic `HexPolyMathlib` lemma:

- `toPolynomial_one`
- `toPolynomial_neg`
- `toPolynomial_sub`

All `@[simp]`, all proved by delegation (no new proof obligations). Builds
clean via `lake build HexPolyMathlib HexPolyZMathlib`.

## Current frontier

Deliverable #1 of issue #7598 listed six lemmas. Only the three above are
well-typed. The other three — `toPolynomial_pow`, `toPolynomial_natCast`,
`toPolynomial_intCast` — reference operations that **do not exist** on the
executable type:

- `Hex.ZPoly` (= `Hex.DensePoly ℤ`) has `Mul`, `Add`, `Sub`, `Neg`, `One`
  instances but **no** `Monoid`/`Pow`, `NatCast`, or `IntCast` instance.
- `HexPolyMathlib.equiv` is only a `≃+*` over `Mul`+`Add` (it does not
  require, and the executable type does not supply, `NonAssocSemiring`/
  `Monoid`), so `map_pow`/`map_natCast`/`map_intCast` cannot fire either.

Confirmed concretely: `p ^ 2`, `((3:Nat) : Hex.ZPoly)`, `((3:Int) : Hex.ZPoly)`
all fail to typecheck (`HPow`/`NatCast`/`IntCast` instance not synthesizable).
So those three lemma statements cannot be written without first adding the
missing executable ring/cast instances — which the issue explicitly forbids
("Do not broaden the task beyond projection/cast normalization").

## Next step

A planner should either (a) narrow #7598 to the three landed lemmas and close

instances to the executable `Hex.DensePoly` layer (an executable-layer change
with its own design + conformance concerns) before the pow/cast projection
lemmas can be stated.

## Blockers

`toPolynomial_pow`/`natCast`/`intCast` blocked on missing executable-layer
instances (see above). Landed as `--partial`.

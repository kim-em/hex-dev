## getD-headed LHS: accepted here

The directive flagged `normalizeWords_get?_getD` as an unknown because
the `getD`-headed `transformedCoeffs_getD_top` was *rejected* in #8134.
Here the `getD`-over-`[i]?` LHS elaborated cleanly under `grind =` and
builds green. The two cases differ in how the index parameter appears:
`normalizeWords_get?_getD`'s LHS `((normalizeWords words)[i]?).getD 0`
mentions both bound parameters `words` and `i`, so grind's pattern can
instantiate everything. (Contrast the rejected `degree` lemma below,
whose RHS-only `d` cannot be reached from the LHS pattern.) The
`getD`-head is therefore not itself disqualifying; what matters is
whether every parameter is reachable from the chosen E-matching
pattern.

## Documented exclusion (directive-as-hypothesis)

The directive listed `degree_eq_of_degree?_eq_some` (L666) as a
definite literal-`=` promotion, but the real build **rejects** it from
`grind =` (hard error). Left `@[simp]`-only:

```
invalid pattern(s) for `degree_eq_of_degree?_eq_some`
  [degree #2]
the following theorem parameters cannot be instantiated:
  d : Nat
  h : p.degree? = some d
```

Root cause: `degree_eq_of_degree?_eq_some {p} {d} (h : p.degree? = some d)
: p.degree = d` has conclusion `p.degree = d`; grind's `grind =`
pattern is the LHS `degree p` (`[degree #2]`), which mentions only
`p`. The RHS-only **data** parameter `d : Nat` cannot be instantiated
from that pattern, so grind rejects it (the hypothesis `h` is dropped
as a consequence). This is the same data-vs-proof dropped-parameter
rule recorded for `p_odd_nat` in the #8148 Montgomery sweep: a lemma
whose conclusion drops a **data** parameter is rejected by `grind =`;
dropping only a **proof** parameter is fine.

General refinement for future sweeps: a conditional `a = b` lemma is
eligible only if every bound *data* parameter appears in the
E-matching LHS `a`. When the witness lives only on the RHS (here `d`),
the lemma is ineligible regardless of the literal-`=` conclusion shape.

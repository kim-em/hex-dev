# Drop silent prime-data fallback (#6598)

## Accomplished

Claimed #6598 and removed the executable silent fallback definition.
The remaining compatibility wrapper `choosePrimeData` now delegates to
`(choosePrimeData? f).get!`, so the no-admissible-prime branch is no longer a
silent `p = 3` / empty-factor fallback.

Added the minimal `Inhabited PrimeChoiceData` instance needed for `Option.get!`
and removed the unconditional `choosePrimeData_prime` theorem, which depended
on the old fallback branch. Existing success-witness facts such as
`choosePrimeData?_prime` remain the proof-facing API.

Tightened bridge call sites that already carry explicit
`choosePrimeData? = some primeData` witnesses:

- `squareFreeCore_irreducible_of_small_mod_singleton` now takes the witness
  shape directly instead of `primeData = Hex.choosePrimeData core`;
- the two `reassemblyExpansionComplete_exhaustive_*` proofs in
  `HexBerlekampZassenhausMathlib/IntReductionMod.lean` use `primeData`
  directly instead of converting back through `Hex.choosePrimeData`.

Updated stale docs/comments so the removed fallback identifier has no remaining
matches.

Verification run:

- `lake build HexBerlekampZassenhaus.Basic`
- `lake build HexBerlekampZassenhausMathlib.IntReductionMod`
- `lake build HexBerlekampZassenhaus`
- `lake build HexBerlekampZassenhausMathlib`
- `lake exe hexbz_bench list`
- `lake exe hexbz_bench verify --filter runFactor`
- grep for the removed fallback identifier
- `git diff --check`
- scoped added-line grep for the project's forbidden identifiers

All passed. Existing dependency warnings include known proof-placeholder
warnings outside this change.

## Current frontier

The full removal of `choosePrimeData` is still blocked by the transitional
total modular compatibility surface left after #6597: `factorSlowFactorsWithBound`
and the product/branch proofs are total and do not carry a local
`choosePrimeData? = some ...` witness in their exhaustive arms.

## Next step

Continue the broader #6584 cleanup by removing the old total modular
compatibility names (`factorSlowFactorsWithBound`, `factorSlowWithBound`,
`factorSlow`) or converting their proof surfaces to explicit Option/witness
forms. Once those are gone, `choosePrimeData` itself should become removable.

## Blockers

None for #6598. The preferred strategy from the issue, deleting
`choosePrimeData` outright, was not feasible in this session because of the
remaining total compatibility definitions described above.

The requested reflection command was unavailable in this environment:
`coordination reflect` is not a recognised subcommand, and no local slash-command
implementation was present.

# 2026-05-04T05:37:49Z — HexPoly python-flint oracle (issue #1987)

## Accomplished

- Extended `HexPoly/EmitFixtures.lean` with the issue-specified
  fixture set:
  - `mul` over `DensePoly Int` at degrees 8 / 16 / 32 / 64, sparse
    and dense, coefficients drawn from a glibc-style LCG modulo
    ~2^16 (8 new cases, on top of the 4 bootstrap small cases).
  - `divmod` over `DensePoly Int` at degrees 8 and 16 with
    exact-division pairs (4 new cases).
  - `gcd` over `DensePoly Rat` at degrees 8 / 16 / 32, constructed
    as `g * a` and `g * b` for known integer factors then cast to
    `Rat` (3 new cases).
- Added a `polyRatValue` helper to `Hex.Conformance.Emit` that
  encodes a list of `Rat` coefficients as parallel `num` / `den`
  arrays, so the gcd result is diffable in JSONL form.
- Extended `scripts/oracle/poly_flint.py` to handle the `gcd` op:
  builds an `fmpq_poly` from the integer-coefficient inputs,
  computes `flint.fmpq_poly.gcd`, and compares to Lean's value
  after normalising both sides to the monic associate.  This
  works around the fact that `Hex.DensePoly Rat.gcd` returns an
  arbitrary rational scalar associate (and `Int.gcd` is structurally
  unsuitable for cross-check because of truncating integer division).
- Regenerated `conformance-fixtures/HexPoly/poly.jsonl` (now 63
  records, 21 result lines).  All cases pass the python-flint
  oracle locally (`python3 scripts/oracle/poly_flint.py --check`
  reports 21 checked, 0 failures).
- Verified the corruption sentinel:
  - Sign-flipping the trailing coefficient in `mul/typical`'s result
    trips the oracle on `mul`.
  - Off-by-one perturbing the constant numerator in `gcd/deg8`'s
    result trips the oracle on `gcd` after monic normalisation.

## Current frontier

`HexPoly`'s oracle surface is now feature-complete for `mul`, `divmod`,
and `gcd` per #1987.  CI's `oracle-pyflint` job already exercises the
fixture stream, so the larger fixture set is picked up automatically
without a workflow change.

## Next step

The remaining oracle work items target other libraries (`HexPolyZ`
#1988, `HexPolyFp` #1989, `HexBerlekamp` #1994, etc.) and reuse the
same emit infrastructure.  The `polyRatValue` helper added here will
likely be reused by any library that needs to cross-check a Q[x]
operation whose Lean answer is only determined up to a scalar
associate.

   `normalizeForFactor_squareFreeCore_monic` exists; it does not.

   `Hex.squareFreeCore_leadingCoeff_pos_of_ne_zero` at
   `HexBerlekampZassenhaus/Basic.lean:8833`) but is not generally
   monic. Concrete counterexample: `f = 2X + 1` gives
   `squareFreeCore = 2X + 1` (primitive, leadingCoeff 2). The wrapper
   `factor_exhaustive_branch_entry_core_zpolyIrreducible_of_henselSubsetCorrespondence`
   (`Basic.lean:8967`) threads `hcore_monic` through to
   `exhaustiveCoreFactorsWithBound_coverage_of_henselSubsetCorrespondence`,
   where monicness is genuinely used by the Hensel correspondence
   proof.

2. **The umbrella's conclusion covers entries from the normalization
   prefix as well as the exhaustive core.** The wrapper at line 8967
   is scoped specifically to `hcore_entry : ∃ raw ∈
   exhaustiveCoreFactorsWithBound..., entry.1 = normalizeFactorSign raw`.
   `factorWithBound_entry_mem_exhaustive_branch_raw` +
   `exhaustiveSlowRawFactorsWithBound_mem_normalization_or_core` give
   only a disjunction: the prefix branch contains the `X`-power
   factors (each irreducible) and the `repeatedPart` factor (which is
   `(X+1)^2` for `f = (X+1)^3`, not irreducible). The sibling
   fast-path arm umbrellas (constant/singleton/quadratic) handle this
   by taking `hcomplete : reassemblyExpansionComplete` as an explicit
   premise; the exhaustive arm has no analog of
   `reassemblyExpansionComplete_constant_of_ne_zero` (#4585 /
   PR #4598) — i.e. no `reassemblyExpansionComplete_exhaustive_of_*`
   substrate that would discharge `hcomplete` from `f ≠ 0` +
   `factorWithBoundUsesExhaustiveBranch`.

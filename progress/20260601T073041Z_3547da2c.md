# Issue #5865 — Conformance.lean `#guard` migration off total `choosePrimeData`

## Accomplished

- Migrated the seven `#guard` blocks on inputs with admissible primes
  (`squareFreeTypical`, `leadingCoeffDivisibleByFive` ×3,
  `extendedCascade2`, plus the two `henselLiftData` adapter tests for
  `squareFreeTypical` and `leadingCoeffDivisibleByFive`) from
  `let data := choosePrimeData ...` to
  `match choosePrimeData? ... with | none => false | some data => ...`,
  preserving the assertion contents on the `some` arm verbatim.
- Replaced the `repeatedRootPoly` prime-selection guard with an
  inverted `match ... | none => true | some _ => false` form.
  `repeatedRootPoly = (x − 1)²` has no admissible prime — its modular
  image stays a perfect square mod every candidate — so the old guard
  was specifically asserting the silent-fallback values (`data.p = 3`,
  `data.factorsModP.size = 1`). The new guard documents the
  no-good-prime contract under the partial API.
- Dropped the `henselLiftData repeatedRootPoly` guard. Hensel lifting
  requires a selected prime; with no admissible prime there is no
  meaningful adapter behaviour to test. The `henselLiftData` adapter
  is still covered by the `squareFreeTypical` and
  `leadingCoeffDivisibleByFive` guards.
- Refreshed the file header bullet from `choosePrimeData` to
  `choosePrimeData?`.
- Trimmed the stale "Before the extended-search fall-through,
  `choosePrimeData` silently used the `fallbackPrimeChoiceData`
  `p = 3` branch" sentence from the extended-cascade docstring,
  matching the cleanup pattern in #5864.

## Blockers

None on this sub-issue. The deviation from the strict "preserve `some`
arm content" wording on the two `repeatedRootPoly` guards was forced
by the input's mathematical structure: `choosePrimeData?` correctly
returns `none`, so the old fallback-specific assertions could not be
preserved unchanged. The PR description documents this.

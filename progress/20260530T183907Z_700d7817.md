# #5842 HO-5d-1b-cont: migrate remaining BZ production callers off total choosePrimeData

Session UUID prefix: `700d7817`

## Accomplished

Migrated the three remaining audit-flagged production callers in
`HexBerlekampZassenhaus/Basic.lean` off the silent total
`choosePrimeData` fallback, completing the executable-side wedge
identified by #5840's progress note:

1. **`exhaustiveSlowRawFactorsWithBound`** — body now calls
   `choosePrimeDataWithFallback` instead of `choosePrimeData`.
2. **`factorSlowFactorsWithBound`** (exhaustive arm) — same migration.
3. **`factorFastFactorsWithBound`** (both `B = 1` and `B > 1` arms) —
   same migration.

The new private definition

```lean
def choosePrimeDataWithFallback (f : ZPoly) : PrimeChoiceData :=
  (choosePrimeData? f).getD (fallbackPrimeChoiceData f)
```

makes the `choosePrimeData?` dispatch and `fallbackPrimeChoiceData`
fall-through visible at every production call site, in line with
`SPEC/design-principles.md` §8 fallback discipline. The bridging
lemma

```lean
theorem choosePrimeData_eq_choosePrimeDataWithFallback (f : ZPoly) :
    choosePrimeData f = choosePrimeDataWithFallback f := rfl
```

establishes definitional equality with the existing total
`choosePrimeData`, so:

- Downstream theorem statements that still name `choosePrimeData ...`
  remain provable (the equality is `rfl`-level).
- The Mathlib bridge files in `HexBerlekampZassenhausMathlib/` continue
  to consume `Hex.choosePrimeData` unchanged — bridge migration is
  scoped to #5817.

For the `factorFastFactorsWithBound_branch` classifier disjuncts (c,
d, e, g, h, i) and a few internal `let primeData := ...` bindings in
the `*_polyProduct_of_some` and `_product_of_exhaustive_branch`
proof chains, the statements / bindings were also switched to
`choosePrimeDataWithFallback` because they appear inside `if_pos
hpred` / `if_neg hpred` chains that need syntactic alignment with the
unfolded body. None of these symbols are referenced outside Basic.lean
or appear in publicly-quoted bridge surfaces, so the change is
internal.

Where a hypothesis named `choosePrimeData` could not be renamed (because
its statement form is part of a private theorem signature consumed by
multiple internal callers), the proof now performs an explicit
`rw [choosePrimeData_eq_choosePrimeDataWithFallback] at h…` before the
subsequent `rw [← h…]` / `rw [if_neg h…]`. This pattern shows up in:

- `factorFastFactorsWithBound_eq_some_of_core_success`
  (the `hnormalized` premise)
- `factorFast_ne_none_of_core_recovery_on_schedule`
  (the `hnormalized` premise)
- `factorWithBound_entry_mem_small_mod_singleton_raw`
  (the `hpred` from `hchoose + hsmall`)
- `factorFastWithBound_product_of_small_mod_branch`
  (the `hpred` from `hchoose + hsmall`)
- `factorFastWithBound_product` (the inner `cases hcore` / `hpred`
  pair on both `B = 1` and `B > 1` arms)

The `exhaustiveSlowRawFactorsWithBound_mem_normalization_or_core` and
`factorWithBound_entry_mem_exhaustive_branch_xPower_or_core_of_reassemblyComplete`
proofs add a single `rw [← choosePrimeData_eq_choosePrimeDataWithFallback]
at h…` after `rw [exhaustiveSlowRawFactorsWithBound] at h…` so the
hypothesis aligns with the existing `choosePrimeData`-shaped
conclusion. The bridge consumer
(`HexBerlekampZassenhausMathlib/IntReductionMod.lean:3181`) reads the
conclusion form unchanged.

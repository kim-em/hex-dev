# #6509 (D2 `choosePrimeData?_none_implies_huge`) — released as replan

## Accomplished

Premise-checked the directive against `HexBerlekampZassenhaus/Basic.lean`
on `main`. The proof route's step 2 ("every candidate prime in the
supported hot-path range was considered and failed `isGoodPrime`") is
false against the implementation, for two compounding reasons:

1. `smallPrimeCandidates` is a curated 10-prime list
   `[3, 5, 7, 11, 13, 17, 19, 23, 31, 71]` (lines 450–460). The
   primes `{29, 37, 41, 43, 47, 53, 59, 61, 67}` are in
   `HotPathCandidates` (each prime, each ≤ `UInt64.word`) but
   absent from the list. The post-prefix walk starts at `73` and
   only advances upward in steps of `+2`
   (`choosePrimeDataWalk?` at line 2013), so these primes are
   never tested by the executable on any input.
2. The walk fuel `choosePrimeDataWalkFuel f := max 256
   (f.toArray.foldl (· + ·.natAbs) (2 * f.size + 1))` (line
   2004) caps the walk well below `UInt64.word = 2^64`. For any
   sane-coefficient `f`, the walk visits microscopically few of
   the ~2^58 primes in `HotPathCandidates`.

Either gap alone breaks the universally-quantified D2
`(p : Int) ∣ (f.leadingCoeff * f.discriminant)` for all
`p ∈ [3, UInt64.word]` prime: the proof route can constrain
`isGoodPrime f p = false` only for primes the executable
actually examines.

Also noted three spelling mismatches the directive will need
when reposted: `ZPoly` carries `Hex.ZPoly.Primitive` (so
`f.Primitive` is fine), does **not** carry an `IsSquareFree`
field (the in-tree spelling is `Hex.SquareFreeRat`), and does
**not** carry a `discriminant` field (the in-tree infrastructure
is `Polynomial.resultant` via
`HexBerlekampZassenhausMathlib/Resultant.lean`).

Posted the concrete-evidence comment
(https://github.com/kim-em/hex/issues/6509#issuecomment-4654790244)
and released the issue with `coordination skip 6509`.

SPEC `Libraries/hex-berlekamp-zassenhaus.md` already contains
the matching escape hatch ("Any worker noticing the walk can
short-circuit before exhausting `HotPathCandidates` should file
the fix as a prerequisite issue, not weaken D2's statement"),
so the report-and-release path is the doctrine-aligned outcome
here.

## Current frontier

#6509 is back in `replan`. The hand-off needs SPEC + executable
alignment, not another worker attempt: either the executable
gets extended to genuinely enumerate every prime in `[3,
UInt64.word]` before reporting `none` (computationally
infeasible at ~10^17 primes, so likely impossible without
abandoning `ZMod64`-bounded scoring entirely) or D2 is restated
in terms of an executable-aligned candidate set.

## Next step

For the next planner: revisit SPEC PR #6580's D2 statement. The
two realistic shapes are (a) replace `HotPathCandidates` in D2's

exponentially-large polynomial coefficients, so D2's
quantification range has to shrink either way.

## Blockers

literally true is infeasible at this layer (would need to
iterate `~10^17` primes per call); a SPEC revision is the
realistic unblock.

## Scope adjustments from the issue's expected signature

The issue's outline of the main theorem used `Irreducible` and
`¬ Associated` hypotheses directly. I formulated the main theorem with the
**direct non-divisibility tail hypothesis** instead. Three reasons:

1. **Soundness for non-monic factors.** The greedy
   `consumeExactPower` relies on `exactQuotient?` agreeing with `ZPoly`
   divisibility, which holds only for monic positive-degree candidates via
   `exactQuotient?_eq_some_of_mul_eq_monic_of_pos_degree`. Requiring
   monicness in the hypotheses pins this down explicitly. Generalising to
   non-monic factors would need a new `divMod_eq_of_mul_eq_of_ne_zero`
   substrate, which is out of scope for this sub-issue.

2. **Bridging stays composable.** The bridge
   `irreducible_not_dvd_of_not_associated` is provided separately, so the
   discharger assembly (sub-issue C) can compose it with the
   Mathlib-side structural fact (sub-issue #4602) to discharge the
   tail-non-divisibility hypothesis pointwise. Keeping the bridge a
   private helper rather than baking it into the main theorem keeps the
   main theorem free of the irreducibility/UFD chain.

3. **Generalising "head doesn't divide product" is its own substrate.**
   The conversion "irreducible `q₁` and `¬ Associated q₁ q_j` for each
   `j` ⟹ `q₁ ∤ ∏ q_j^{e_j}`" requires UFD/prime-like reasoning over
   `ZPoly`. This is doable on the Mathlib side (sub-issue #4602)
   leveraging `UniqueFactorizationMonoid.radical` and friends; baking it
   into the Mathlib-free helper would duplicate that infrastructure.

The main expansion theorem's hypothesis bundle (monic + positive degree +
tail-non-divisibility + decomposition + fuel) matches what the
discharger-assembly sub-issue can hand it once the sibling Mathlib
structural fact lands.

### New SPEC files

- `SPEC/Libraries/hex-resultant.md` — polynomial resultant +
  discriminant via the subresultant pseudo-remainder sequence
  (Brown 1971; Collins 1967). Mathlib-free, depth 1, deps
  `[HexPoly]`. Asymptotically `O(n·m)` polynomial-pseudodivision
  steps, far better than `O((n+m)³)` for the Sylvester+Bareiss-
  determinant alternative.
- `SPEC/Libraries/hex-resultant-mathlib.md` — **scope-limited**
  Mathlib bridge proving "subresultant zero ↔ common root" via
  subresultant chain analysis (~600 lines of formalisation). The
  full `Hex.resultant = Polynomial.resultant` bridge is deferred
  as a future Mathlib upstream contribution; the narrower property
  suffices for `HexNumberField`'s downstream needs (specifically,
  `α + β` being a root of `resultant_y(p_α(y), p_β(t−y))`).
- `SPEC/Libraries/hex-number-field.md` — combined library for
  `NumberField p x` (an element of `ℚ(α)` indexed by a complex
  simple root) and canonical `AlgebraicNumber` (representing any
  `α ∈ ℂ_alg` by `(minimal polynomial, SimpleRoot)`). Deps
  `[HexPolyZ, HexRoots, HexResultant, HexBerlekampZassenhaus]`.
- `SPEC/Libraries/hex-number-field-mathlib.md` — Mathlib bridge:
  `NumberField p x ≃+* AdjoinRoot p` under irreducibility,
  bijection of `AlgebraicNumber` with `ℂ_alg`, arithmetic
  correctness theorems via the primitive-element theorem and
  `commonField` semantics.

### Notable design decisions captured in the SPECs

- **Subresultants over Sylvester+Bareiss for resultant.** The
  subresultant chain is asymptotically faster (`O(n·m)` polynomial-
  pseudodivision steps vs `O((n+m)³)` matrix operations), and
  doesn't need a `HexMatrix` dep. The decision was driven by user
  pushback against the initial Sylvester+Bareiss design.
- **Scope-limited Mathlib bridge** for resultant. `subresultant_zero
  _iff_common_root` (~600 lines) suffices for our use cases; the
  full `resultant_eq` (~1500–2000 lines) is deferred. The narrower
  proof goes via subresultant *chain analysis* (pseudo-division
  preserves common roots forward and backward, chain computes a
  gcd, gcd is constant iff no common roots), bypassing the
  Sylvester-determinant connection entirely.
- **`Hex.ZPoly.Irreducible` as a `class`.** Necessary because
  Mathlib's `Fact` wrapper (the standard idiom for typeclass-driven
  prop arguments) is unavailable in our Mathlib-free layer. This
  contradicts the project's usual "irreducibility is a term-level
  fact" convention, but the convention applies when irreducibility
  is part of a type's identity (cf. `HexGfqField.FiniteField`); for
  `NumberField.Inv`'s case, it's an ambient assumption used by an
  operation, where a class is appropriate.
- **`AlgebraicNumber` carries no `coeffs` field.** The simple root
  itself is the algebraic number — no coefficient data needed. This
  gives genuinely canonical structural equality
  (`AlgebraicNumber` = canonical form of `ℂ_alg`).
- **`commonField` is public API**, returning `Σ' (r : ZPoly) (γ :
  SimpleRoot r), Hex.ZPoly.Irreducible r × NumberField r γ ×
  NumberField r γ`. Users who want explicit field-embedding
  computation get it; those who want pure `α + β` semantics use the
  arithmetic ops which call `commonField` internally.
- **Threading pattern for `SimpleRoot.out`.** `out` is canonical and
  decidable on the equivalence class, but expensive when the
  underlying representative is at low precision. `SimpleRoot.refine
  target` returns a propositionally-equal SimpleRoot whose internal
  representative is at higher precision; subsequent `out` calls
  short-circuit refinement. `AlgebraicNumber` operations follow
  this pattern: each op returns a refreshed value with refined
  internal representative, callers store/forward.

# Phase 6: tune grind automation for UInt64 wide-word reconstruction (issue #6168)

## Accomplished

- Added `@[grind =]` to the three product-reconstruction equations in
  `HexArith/UInt64/Wide.lean`:
  - `mulHi_mulLo`
  - `mulLo_add_mulHi`
  - `mulFull_snd_add_fst`
  Each has a specific LHS pattern (high/low product reconstruction) that
  rarely fires outside genuine wide-word reasoning, so the annotation is
  conservative.
- Simplified two proof fragments in `HexArith/Montgomery/Context.lean`
  (`twoWordProduct_lt_p_word` and `redc_mulFull_repr_word`) by replacing
  explicit `exact UInt64.mulFull_snd_add_fst a b` / `rw […]` invocations
  with a single `grind`, demonstrating the annotation is doing useful
  work.
- Left `toNat_addCarry` and `toNat_subBorrow` unannotated. Both are
  stated over a destructuring `let (s, cout) := …`, which does not
  give grind a clean E-matching pattern. The existing consumers
  destructure via `cases hpair : addCarry … with | mk lo c1 => …`
  before invoking the lemma, so they do not need grind anyway.

## Audit notes (kept here in lieu of a journal in source)

- One naive attempt to replace the `hnum` derivation inside
  `redc_u_spec` (`HexArith/Montgomery/Redc.lean`) with `grind`
  triggered a 118 s search and a kernel "deep recursion" failure
  even with `mulHi_mulLo` in context. The goal is nonlinear over
  `m.toNat * p.toNat`, which is exactly where grind's search
  blows up. That confirms the issue's caution about avoiding
  annotations that cause loops, and is why the existing
  `omega`-based proof in `redc_u_spec` is kept intact.
- Build timings on `HexArith.Montgomery.Context` were 992 ms / 937 ms
  on the two intermediate builds after introducing `by grind`; full
  `HexArith` rebuild stayed green with no slowdown elsewhere.

# Diagnosed #8290 as premise-unsound (dilation covariance)

## Accomplished

Premise-checked #8290 (core-coordinate forward cut: `SupportShortVectorData`
producer for `bhksLatticeBasis core …`) and found the decomposition unsound.
Posted a full diagnosis comment with a concrete counterexample and
`coordination skip`ped the issue to `replan`. No Lean changes; no sorry/axiom.

**The finding.** The executable builds `bhksLatticeBasis core d.p d.k
d.liftedFactors` with `f = core` (raw non-monic) but lifted factors `gᵢ` that are
monic Hensel factors of `(toMonic core).monic`. `toMonic` is the *dilate*-monic
transform `D(x) = ℓ^{n-1}·core(x/ℓ)` (`transformedCoeffs`, `Basic.lean:1430`), so
`gᵢ = σ_ℓ(h)` is a dilation of an integer factor `h | core`, not a scalar
multiple `c·h`. Consequences:

- Piece (1) "scale-invariance `phi(core, c·h) = phi(core, h)`" is both vacuous
  (`phi` is `0` for non-monic args) and mis-models the obstruction: the true
  relation is `supportProduct ≡ σ_ℓ(h)` (dilation), not `c·h` (scalar).
- Piece (2) "core aggregate-residue" rests on `cldQuotientMod_congr_mul_derivative`
  (`CLDColumnBound.lean:607`), which requires `gᵢ ∣ core (mod p^a)`. The `gᵢ`
  divide the *dilated* `toMonic core`, not `core`, so the hypothesis is false.
- Counterexample: `core = 2x²−3x+1 = (x−1)(2x−1)`, `toMonic core = (x−1)(x−2)`.
  Factor `x−1` maps to monic-coordinate `x−2`, which does not divide `core`
  (`core(2)=3`). The executable column `cldQuotientMod core (x−2) = 2x+1`, but the
  genuine raw CLD column for `x−1` is `core/(x−1) = 2x−1`. So `2x+1 ≠ 2x−1`.

The real obstruction is structural dilation covariance of the CLD column under
`x ↦ ℓ·x` (the #7479-class dilation-automorphism obstruction), which is soundness
substrate, not a thin producer. The raw-core lattice is only validated downstream
by the `Array.polyProduct candidates == f` product check, not a provable cut
radius bound.

Cross-checked with an independent second opinion (Codex), which reproduced the
counterexample and agreed with the skip.

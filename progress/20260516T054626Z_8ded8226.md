# HexPoly Phase 6: executable division/GCD surface review (#4408)

## Accomplished

- Audited the executable division / monic-division / field-division /
  GCD / extended-GCD surface at the top of `HexPoly/Euclid.lean`
  (≈ lines 1–1098) against `PLAN/Phase6.md`,
  `SPEC/Libraries/hex-poly.md`, and the documentation expectations
  in `SPEC/design-principles.md`.
- Cross-checked the two downstream callers named in the issue
  (`HexPolyFp/Quotient.lean` and `HexBerlekamp/RabinSoundness.lean`).
  Both stay at the public API level: `Monic`, `modByMonic`, `gcd`,
  `xgcd`, `leadingCoeff`, `modByMonic_eq_mod`, `modByMonic_zero`,
  `gcd_dvd_left/right`, `xgcd_bezout`. Neither caller unfolds the
  private `divModArrayAux` / `xgcdAux` / `arrayDegree?` /
  `subtractScaledShift` internals. The public surface is sufficient
  for them as-is.
- Landed missing docstrings on 16 public theorems and on the `Div`
  and `Mod` notation instances. The previously documented decls
  (`divModArray_scaleLead_congr`, `divMod_eq_zero_self_of_size_zero_core`,
  `divModMonic_eq_divMod_of_monic_core`,
  `divMod_remainder_degree_lt_of_pos_degree`,
  `divModMonic_eq_divMod_of_monic`, `mod_eq_self_of_degree_lt`,
  `mod_mod_of_not_pos_degree`, `mod_degree_lt_of_pos_degree`,
  `zero_mod_eq_zero_core`, `modByMonic_zero`) were left as-is.
- Added `@[simp]` to two unconditional normalization lemmas:
  `gcd_zero_zero` (`gcd 0 0 = 0`) and `mod_mod`
  (`(p % q) % q = p % q`). Both are classic simp shapes from the
  Mathlib polynomial API. No `@[simp]` tags existed in this file
  previously even though `HexPoly/Dense.lean` and
  `HexPoly/Operations.lean` already follow that convention.

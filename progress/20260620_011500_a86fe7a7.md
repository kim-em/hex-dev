# Phase 6 grind coverage — HexModArith/Ring.lean arithmetic-identity cluster (#8135)

## Accomplished

Audited the nine `@[simp]`-only `ZMod64` arithmetic-identity lemmas in
`HexModArith/Ring.lean` (L444–L508) for `@[simp, grind =]` promotion.
Promoted the three that elaborate as valid E-matching rewrite rules:

- `add_zero` (L444) — `a + 0 = a`
- `zero_add` (L452) — `0 + a = a`
- `pow_succ` (L502) — `a ^ (n + 1) = a ^ n * a`

## Documented exclusions

The other six were **rejected by `grind =` elaboration** (hard build
error) and left `@[simp]`-only:

- `mul_zero` (L460) — `a * 0 = 0`
- `zero_mul` (L467) — `0 * a = 0`
- `mul_one` (L474) — `a * 1 = a`
- `one_mul` (L481) — `1 * a = a`
- `pow_zero` (L488) — `a ^ 0 = 1`
- `pow_one` (L495) — `a ^ 1 = a`

Root cause: `ZMod64 p` carries a `Lean.Grind.Semiring` instance
(`HexModArith/Ring.lean:441`). grind's ring normalizer collapses each of
these LHSs to a ground term or a bare bound variable — `a * 1`, `1 * a`,
`a ^ 1` → `a`; `a * 0`, `0 * a` → `0`; `a ^ 0` → `1` — so no usable
pattern application remains. The errors are *"invalid pattern,
(non-forbidden) application expected #0"* and *"invalid pattern(s) for
`pow_zero` … parameters cannot be instantiated: a"*. These identities are
already derivable by grind from the `Semiring` instance, so the
exclusions lose no automation. The additive identities (`add_zero`,
`zero_add`) survive because grind has no additive-identity normalization
for this type, leaving a genuine `HAdd` application as the pattern;
`pow_succ`'s LHS `a ^ (n + 1)` mentions both `a` and `n` in a real
application.

This is the directive-as-hypothesis path: the issue listed all nine as
eligible, but six turned out ineligible against the real build, and the
issue explicitly sanctions documented exclusions as a valid outcome.

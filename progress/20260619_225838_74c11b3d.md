# Whole-sweep audit of Phase 6 grind-automation coverage (#8116)

Review session. Audited the result of the multi-PR Phase 6 sweep that
promotes equation-shaped public `@[simp]` lemmas to `@[simp, grind =]`.
Read-and-verify only; outcome is **no annotation changes needed** — the
sweep is internally consistent on the dimension that matters (no looping /
ineligible `grind =` rule), and the deviations found are well-formed.

## Accomplished

Parsed every `grind =` declaration in `Hex*` (800 genuine `grind =`,
distinct from 49 `grind =>` forward-rules, 5 `grind →`, 4 `grind ←`,
103 bare `grind`, 8 `grind only`) and classified each conclusion's
top-level connective, respecting paren/bracket depth.

### Deliverable 1 — no ineligible promotion: PASS

Every `grind =` lemma has either an `=` or `↔` conclusion. No `grind =`
carries a top-level `∧`, `∨`, `≠`/`Ne`, `≤`, `<`, `Associated`, or
`let`-binder conclusion. (`grind =>` is a *different* attribute; the
`0 < …`, `… ∧ …`, and top-level-`let` conclusions all live on `grind =>`,
not `grind =`, so they are out of the equational sweep's scope. Two
`coeff_shift*` lemmas surfaced as false positives — their `<` is inside an
`if k < n then …` on the RHS of a genuine `lhs = rhs`.)

Three `grind =` lemmas have an `↔` conclusion, which the sweep's *stated*
mechanical heuristic lists as "leave `@[simp]`-only":

- `isZero_iff_eq_zero` (`HexGF2/Basic.lean`, `@[simp, grind =]`,
  promoted by #7977): `p.isZero = true ↔ p = 0`
- `mem_polysBelowDegree_succ_iff_degree_getD_lt`
  (`HexPolyFp/Enumeration.lean`, `@[simp, grind =]`, promoted by #7986):
  `… ↔ f.degree?.getD 0 < d + 1`
- `mem_polysBelowDegree_iff_degree_getD_lt`
  (`HexPolyFp/Enumeration.lean`, bare `@[grind =]`, added by #7986)

**Not reverted, by design.** The directive's premise that an `↔`
conclusion is *ineligible* for `grind =` does not hold on this toolchain
(`v4.30.0-rc2`): `grind =` accepts `↔` and treats it as a rewrite. Three
lines of evidence:

1. All three compile under whole-repo merge-gating CI on green `main`. If
   `grind =` rejected `↔`, the build would be red.
2. The codebase already carries ~14 *deliberately authored* bare
   `@[grind =]` iff lemmas independent of the simp-sweep — the entire
   `HexLLL/Basic.lean` `*_memLattice_iff` family (13) plus
   `HexGFqField/Basic.lean`'s `toQuotient_inj`. Hand-authored `grind =`
   on `↔` is an established, intentional pattern here, not an accident.
3. All three rewrite to a strictly simpler RHS (collapse `isZero` to an
   equality; reduce a membership to a degree inequality), so they
   terminate.

The sweep's "equation-only" rule is therefore a conservative *mechanical*
heuristic for the automated pass, not a project correctness rule.
Reverting these would weaken `grind` for zero correctness benefit, so per
the `directives-are-hypotheses` doctrine the finding is recorded rather
than mechanically actioned.

### Deliverable 2 — no missed eligible lemma: spot-checked

Named exclusions confirmed intentional:

- `HexModArith/Ring.lean` ring-axiom lemmas (`add_zero`, `zero_add`,
  `mul_zero`, `zero_mul`, `mul_one`, `one_mul`, `pow_zero`, `pow_one`,
  `pow_succ`) stay `@[simp]`-only because `grind` already derives them
  from the `CommRing (ZMod64 p)` instance (per closed #8014). Contrast
  `HexGF2/Field.lean`, which *does* promote the same axiom shapes for
  `GF2n` — a coherent type-dependent distinction (the Mathlib-free
  `GF2n` does not hand `grind` those rules via a typeclass).
- `HexArith/ExtGcd.lean` has no `@[simp]`-only equation candidates left
  (its `_spec`/`_bezout` lemmas are not `@[simp]`).

~50 public `@[simp]`-only equation lemmas remain across already-swept
files (largest: `HexGF2/Field.lean` ×21). These are **not a uniform
oversight**: most are recursion-step / list-fold equations whose RHS
*duplicates* the recursive call (e.g.
`frobeniusIter_succ : frobeniusIter a (k+1) = frobeniusIter a k *
frobeniusIter a k`), a principled reason to keep them off `grind =`
(E-graph doubling). Recursion-step equations are split codebase-wide
(23 promoted `_succ`/`_cons`/`_nil` vs 13 `@[simp]`-only), so a residual
genuine-miss subset may exist, but separating doubling-RHS exclusions
from true misses requires a per-module build to confirm no grind-loop —
that is per-file feature work, which planners are already dispatching one
file at a time (#8104 `RankSpanNullspace.lean`, #8112 `GF2q.lean`). No
mass promotion in this audit PR; the candidate files are listed here for
that ongoing per-file stream rather than a competing catch-all issue.

### Deliverable 3 — no grind-loop / elaboration regression: PASS

Zero Lean files changed, so build health is inherited from green `main`.
CI is whole-repo merge-gating, so any `grind =` rule that looped a
downstream goal would already have turned `main` red; it is green.
`scripts/check_dag.py` exit 0.

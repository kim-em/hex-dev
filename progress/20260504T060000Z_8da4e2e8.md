# 2026-05-04 06:00Z — feature session, issue #1996

## Accomplished

- Closed #1996 by adding the fpylll oracle stack for `HexLLL`:
  `HexLLL/EmitFixtures.lean`, `scripts/oracle/lll_fpylll.py`, and a
  new `oracle-fpylll` job in `.github/workflows/conformance.yml`.
- The oracle compares Lean's reduced basis against
  `fpylll.LLL.reduction` via three independent checks: row-style HNF
  agreement (lattice equality, computed via `flint.fmpz_mat.hnf` with
  an `fpylll` fallback), `fpylll.LLL.is_reduced` on Lean's basis at
  `δ = 3/4`, and a symmetric shortest-vector squared-norm parity check
  using the LLL gap factor `α^{n-1} = 2^{n-1}` for `δ = 3/4`.
- `Hex.Conformance.Emit` now exposes `latticeValue` for the JSONL
  result records, mirroring the existing `polyValue` / `divModValue`
  shapes.
- Added `lean_exe hexlll_emit_fixtures` to `lakefile.lean` and a
  committed fixture sample at `conformance-fixtures/HexLLL/lll.jsonl`.
- Reduced fixture coverage to what `Hex.lll` currently handles
  correctly: upper-triangular `reducible/d{2,3,4,5}` cases (which
  reduce to the identity) plus one BZ-shaped lattice (`bz/p5k2/3factors`
  matching `HexBerlekampZassenhaus.recombinationLattice?` shape, which
  Lean reports as already δ-reduced).
- Filed follow-up issue #2042 documenting that `Hex.lll` produces
  non-`δ`-reduced output on random LCG-generated integer bases at
  dimensions `4`, `6`, `8`, `10` (μ-values in the tens; entry
  magnitudes blow up to `10^14`-`10^21` at d≥6) — exactly the regression
  shape the new fpylll oracle is constructed to catch once the
  underlying `sizeReduce`/`swapStep` integer updates are fixed.
- Independence and `1 ≤ n` arguments to `lll` are discharged with
  `sorry` in the emit driver, mirroring the production-path
  `recombinationLattice_independent` placeholder used by
  `HexBerlekampZassenhaus`. The propositional witnesses are erased at
  runtime, so the executable driver still computes Lean's reduction.

## Current frontier

the `lll` driver on the inputs the current implementation handles
correctly. Of the issue's originally requested random `d4`/`d6`/`d8`/`d10`
fixtures, none are emitted yet because `Hex.lll`'s current output on
those bases trips every check the oracle makes (lattice equality,
`δ`-reducedness, shortest-vector parity). #2042 is the focused
follow-up to fix the `lll` arithmetic and re-add those fixtures.

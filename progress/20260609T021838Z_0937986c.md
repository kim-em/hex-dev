# HexLLL provider hook (#6614)

## Accomplished

Added the HexLLL optional-provider hook:

- wired `extern_lib hexlllffi` in `lakefile.lean`;
- added `HexLLL/ffi/lean_hexlll_provider.c`, which caches a first-call
  `dlsym(RTLD_DEFAULT, "lean_fplll_lll_reduce")` result and forwards the
  provider ABI when present;
- exposed `LLLProvider.providerAvailable` and `LLLProvider.providerReduce`;
- added Lean-side flat ABI validation for status, dimensions, inverse flag,
  and exact length before slicing;
- added IO diagnostics for absent/provider-error/rejected/accepted outcomes;
- added `hexlll_provider_probe` plus CI probe steps for absent and
  preloaded-present provider-symbol cases on Linux and macOS.

Verification run:

- `lake build HexLLL`
- `lake build hexlll_provider_probe`
- `.lake/build/bin/hexlll_provider_probe absent`
- `DYLD_INSERT_LIBRARIES="$PWD/.lake/build/hexlll_probe_provider.dylib" .lake/build/bin/hexlll_provider_probe present`
- `python3 scripts/check_dag.py`
- `python3 scripts/conformance_targets.py --check`
- `git diff --check`
- forbidden-token scan for `sorry`, `axiom`, and `native_decide` in touched files
- `lake build`

All passed. `lake build` replayed pre-existing style warnings in
`HexGramSchmidt`/`HexGFqRing`.

Attempted `/reflect`; this execution surface has no such command
(`zsh: no such file or directory: /reflect`).

## Current frontier

The hook and ABI validation are in place. `lll` itself is unchanged because
candidate certification and dispatch are explicitly out of scope for this
issue and the checker is not present in `HexLLL` yet.

## Next step

Implement the certificate checker and dispatch layer so `lll` can call
`LLLProvider.tryReduce`, reshape validated candidates, certify them, and move
the accepted diagnostic increment to post-certification acceptance.

## Blockers

None for the provider hook. The CI "present" probe uses a preloaded symbol stub
to exercise `RTLD_DEFAULT` resolution without adding a Lake dependency on the
external provider package.

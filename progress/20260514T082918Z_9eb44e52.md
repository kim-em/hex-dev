## Accomplished

Generalised `reassemblePolynomialFactors` in `HexBerlekampZassenhaus/Basic.lean`
to expand each supplied core factor `q` to its multiplicity in
`squareFreeCore * repeatedPart`, so the recorded `Factorization` carries
correct exponents for higher-multiplicity inputs (cubes and beyond), not just
the `repeatedPart = squareFreeCore` square case fixed in #3951.

Mechanics:

- New executable helpers
  - `exactQuotient?` (moved from later in the file so it precedes the new
    helpers; also moved its `exactQuotient?_product` correctness lemma)
  - `consumeExactPower target candidate fuel : ZPoly × Nat` greedily peels
    `candidate^k` out of `target` via repeated exact division, with invariant
    `candidate^multiplicity * residual = target`.
  - `expandRepeatedPartFactorsAux` / `expandRepeatedPartFactorArray` fold
    `consumeExactPower` over the supplied `coreFactors` against `repeatedPart`,
    emitting one copy of each `q` per successful exact division, with the
    bookkeeping invariant `polyProduct emitted * residual = repeatedPart`.
- `reassemblePolynomialFactors` now dispatches on whether the expansion fully
  consumes `repeatedPart`:
  - Full consumption (residual = 1) → emit `xPower ++ expanded ++ coreFactors`.
    The expanded copies turn each `q` from `coreFactors` into `q^{m_q}` after
    `collectFactorMultiplicities` regroups duplicates.
  - Partial consumption → fall back to the original
    `polynomialNormalizationPrefixFactors d ++ coreFactors` shape; this only
    occurs when the BZ pipeline emitted the raw, possibly reducible
    square-free core as a single core factor.

Product proof:

- `polyProduct_reassemblePolynomialFactors` now states the unified, non-
  conditional identity
  `polyProduct (reassemble d cf) = shift xPower d.repeatedPart * polyProduct cf`.
  The two branches collapse to the same shape because in both
  `polyProduct expanded = repeatedPart` (residual=1 case) or
  `polyProduct (repeatedPartFactorArray d.repeatedPart) = repeatedPart`
  (fallback case).
- `reassemblePolynomialFactors_product_eq_input` correspondingly drops its
  `hdup` case split.

Membership bookkeeping:

- New `reassemblePolynomialFactors_mem` shows every emitted factor is either
  in `polynomialNormalizationPrefixFactors d` or in `coreFactors`, with the
  expansion path lemma `expandRepeatedPartFactorArray_mem` proving emitted
  copies are members of the supplied `coreFactors`.
- New `reassemblePolynomialFactors_normalizeFactorSign_of_ne_zero` and
  `reassemblePolynomialFactors_shouldRecord_of_ne_zero` lift caller-supplied
  per-factor properties through the reassembly, collapsing the per-call-site
  `unfold + by_cases hdup + Array.toList_append + List.mem_append` cascades
  into one or two `refine ... ?_; intro c hc; exact ...` calls.
- Special-case `reassemblePolynomialFactors_singleton_one_eq` for the
  constant branch (`coreFactors = #[1]`) collapses both reassembly branches
  to `polynomialNormalizationPrefixFactors d ++ #[1]`, preserving the
  existing `_append_one` filtering proof shape.

Downstream proof updates (all in `HexBerlekampZassenhaus/Basic.lean`):

- `factorSlowWithBound_product_of_constant_branch`,
  `factorFastWithBound_product_of_constant_branch`: now use
  `reassemblePolynomialFactors_singleton_one_eq` instead of unfolding
  `reassemblePolynomialFactors` and case-splitting on the old `hdup`.
- `factorSlowWithBound_product_of_quadratic_branch`,
  `factorSlowWithBound_product_of_exhaustive_branch`,
  `factorFastWithBound_product_of_squareFreeCore_emit`,
  `factorFastWithBound_product_of_quadratic_branch`,
  `factorFastWithBound_product_of_core_success_branch`: each collapses to
  `reassemblePolynomialFactors_normalizeFactorSign_of_ne_zero` /

  core-factor lemma (`quadraticIntegerRootFactors?_normalizeFactorSign`,
  `exhaustiveCoreFactorsWithBound_normalizeFactorSign`, etc.).

# HexGramSchmidt Phase 6 grind annotation pass (#7960)

## Accomplished

Added `@[grind =]` to the equation-shaped public characterising lemmas
of `HexGramSchmidt/Basic.lean` (the `GramSchmidt.Rat.*` API and its
`GramSchmidt.Int.*` mirror). The annotation count on the file goes from
4 to 34 `grind`-visible lemmas.

Annotated (Rat + Int mirror, narrow LHS triggers keyed on
`basis (rowAdd/rowSwap …)` / `entry (coeffs (rowAdd/rowSwap …)) …`):

- `basis_zero`, `coeffs_diag` — augmented existing `@[simp]` to
  `@[simp, grind =]`, completing the diagonal/triangular characterization
  alongside the already-`@[grind =]` `basis_orthogonal` / `coeffs_upper`.
- `basis_rowAdd`, `basis_rowSwap_of_before`, `basis_rowSwap_of_after`,
  `basis_rowSwap_adjacent_prev` — basis invariance/re-orthogonalization
  equalities (lattice-preservation / size-reduction).
- `coeffs_rowSwap_adjacent_lower_prev`/`_lower_curr`/`_before`/
  `_after_low`/`_after_high`, `coeffs_rowAdd_lower`/`_pivot`/
  `_above_pivot`/`_other_row` — coefficient-update equalities.

## Decisions

Deliberately **not** annotated:

- `basis_decomposition` — LHS is the bare input row `b.row i`; as a
  `grind =` rewrite it would fire on every input row and expand into the
  basis-plus-combination form, risking blowup in any consumer reasoning
  about `b`. Too-general trigger.
- `coeffs_lower_projection` / `_comm` — `if`-shaped RHS and a `_comm`
  duplicate; two competing rewrites for the same `entry (coeffs b) i j`
  LHS.
- `basis_rowSwap_adjacent_curr`, `coeffs_rowSwap_adjacent_pivot` —
  `let`-bound RHS plus a `≠ 0` hypothesis; poor rewrite-rule shape.
- `basis_span` — `↔` with an inner `∀ v` binder, not a rewrite.

No in-file proof shortened: the file's ~113 `by grind` sites all live in
the `private` helper section (lines 15–2985), which **precedes** the
public API and cannot consume it. The Int wrappers are minimal
`simpa … using GramSchmidt.Rat.<lemma>` bridges across `castIntMatrix`;
converting them to `grind` would be contrived and fragile, which the
directive explicitly warns against. Justification for the annotations is
therefore **downstream `grind` availability** — the characterising
row-operation equalities become visible to the `grind` calls in
`HexGramSchmidt/Update.lean` and the CI-gated `*Mathlib` consumers
(`HexGramSchmidtMathlib/{Int,Update}.lean`, `HexLLLMathlib/Independent.lean`).

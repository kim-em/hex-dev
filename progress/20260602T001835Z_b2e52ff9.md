## Reverted candidates (audit discipline)

Per the issue's "remove the annotation if it loops, bloats search, or
breaks any unrelated proof" rule, the following candidates from the
issue body were tried and reverted:

- `@[simp]` on `mod_eq_divMod`, `modByMonic_eq_divModMonic`: the
  rfl-level unfolders unfold `p % q` to `(divMod p q).2` everywhere,
  which collides with proofs that pattern-match on `%`.
- `@[simp]` on `modByMonic_eq_mod`: the rewrite
  `modByMonic _ _ → _ % _` clashes with downstream `simpa using ...`
  patterns in `HexPolyFp/Quotient.lean` that supply
  `polyCRT_modByMonic_snd` as the using term — simp normalizes both
  sides differently and produces a type mismatch.
- `@[simp]` on `modByMonic_zero`: makes
  `simpa using (modByMonic_zero g hmonic).symm`
  reduce the term to `True`, breaking
  `HexPolyFp/Quotient.lean:961`.
- `@[grind =]` on `mul_assoc_poly`, `add_assoc_poly`,
  `mul_add_right_poly`, `mul_add_left_poly`, `sub_eq_add_neg_poly`:
  the associativity and distribution rewrites multiplied
  `HexBerlekampZassenhaus.Basic` build time from ~2 min to >4 min
  (grind explosion) without a corresponding compensating
  simplification at downstream call sites.
- The `dvd_*` cluster was not annotated; the issue body marked it as
  conditional on grind not exploding, and the assoc/distribution
  result above shows that path is fragile.

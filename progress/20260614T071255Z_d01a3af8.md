## Current frontier / Blockers

Deliverable (b)'s *final* bullet — "Conclude `p^(k*d) ∣ resultant
inputPolynomial auxiliaryPolynomial`" — is **not** closable as stated, and
this is a genuine premise gap, not a missing-effort gap:

- The CLD syzygy that `cldQuotientMod_congr_mul_derivative` / `auxCutBridge`
  supply is *per single factor*: `g_i * cldQuotientMod_i ≡ input * g_i'`.
  My bridge consumes exactly this, with the syzygy partner `g` being one
  factor's `cldQuotientMod`.
- But `auxiliaryPolynomialWithCorrections` is the *combination*
  `Σ_i vec i • cldCoeffs i − corrections` over **all** lifted factors. The
  single-factor syzygy `auxiliary*q ≡ input*q'` for a selected `q = g_{i₀}`
  would require `auxiliary ≡ cldQuotientMod_{i₀} (mod p^k)`, which holds
  only when the bad vector is the unit `e_{i₀}` (and corrections vanish).
  For a general bad vector it is false (take `vec = e_{i₁}`, `i₁ ≠ i₀`).
- The exponent in the downstream (`k * localFactorDegree`, a *single* local
  factor's degree) is consistent with `q` = one factor, *not* a product —
  reinforcing that the conclusion targets the single-column case.

So the syzygy route closes `resultant input (cldQuotientMod_q)`, but
reconciling `cldQuotientMod_q` with the bad-vector `auxiliaryPolynomial`
(mod `p^k`) is a distinct modular-reduction step. It belongs to the
downstream wiring (#6949), which the issue already says to leave there.

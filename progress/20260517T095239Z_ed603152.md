### Signature deviation from the issue body

The issue's narrowed-scope deliverable specified `hcore_monic` (Gap 1)
and `hcomplete` (Gap 2) as the two explicit shim premises. During
implementation I found three additional substrate gaps that the
issue's discharger map labelled "derivable" but which are not closed
in the current substrate:

* **Gap 3a — `hB_ne_zero`**: `defaultFactorCoeffBound f ≠ 0` for
  `f ≠ 0`. No `defaultFactorCoeffBound_pos_of_ne_zero` lemma exists
  in `HexPolyZ/Mignotte.lean`. (True in principle:
  `f ≠ 0 → ∃ i, (f.coeff i).natAbs > 0 → coeffNormSq f ≥ 1 →
  coeffL2NormBound f ≥ 1 → defaultFactorCoeffBound f ≥
  mignotteCoeffBound f 0 0 = 1`.)
* **Gap 3b — `hd_modulus`**: `2 ≤ primeData.p ^ precisionForCoeffBound
  (defaultFactorCoeffBound f) primeData.p`. Derivable from
  `primeData.p ≥ 2` (via `choosePrimeData?_prime`) together with
  Gap 3a (which would give `defaultFactorCoeffBound f ≥ 1`, hence
  `precisionForCoeffBound ≥ 1`).
* **Gap 3c — `hprecision`**: the wrapper's core-shape Mignotte
  invariant `2 * defaultFactorCoeffBound core < d.p^d.k`. This was the
  open monotonicity question tracked by #4539 (closed without
  resolution; recommendation in PR #4541 was the abstract-bound
  refactor of `centeredLift_scaledLiftedFactorProduct_eq_of_mignottePrecision`).
  The closed `precisionForCoeffBound_spec` gives the *outer*-shape
  variant; bridging to the core shape needs either #4539 or the
  abstract-bound refactor.

Following the precedent set by `hcore_monic`/`hcomplete` (and earlier
by the fast-path arm umbrellas #4564 / #4565 / #4571 / #4575 which
take `hcomplete` explicitly), I lifted these three additional gaps to
explicit shim premises with a docstring that names each substrate
issue and points at the corresponding upstream work. The capstone
#4170 will need to thread all five shim premises until each substrate
gap closes; a small follow-up issue can then drop them.

## Current frontier

- The planner phase is rock-solid; the worker phase is gated on two
  GitHub-side / Claude-Code-side issues that are not bugs in pod or
  bubble:
  1. **GitHub REST/GraphQL `?labels=` returns empty** for these
     just-created labels even though the issues clearly carry them.
     Toggling the label off/on did not flush the index. This blocks
     pod's dispatcher: `coordination queue-depth` returns 0, so
     `dispatch_queue_balance` keeps picking the filling type (`plan`)
     instead of draining (`work`). Either GitHub's index needs more
     time to propagate (>10 min observed and counting), or there is a
     scope/oauth quirk specific to private repos with sub-second
     label attachment. Reproducible from host gh, bubble's gh, and
     raw curl.
  2. **Claude Code's Bash tool runs from a different cwd than claude's
     process cwd.** Pod's wrapper does `cd /home/user/<repo> && exec
     claude …` so `claude` itself is in the right cwd, but the agent's
     Bash tool calls land in `/home/user`. Coordination's repo
     auto-detect (`gh repo view --json nameWithOwner` from cwd) fails
     and the agent has to fall back to passing `--repo` explicitly,
     which it does, just clunkily. The planner worked around this; a
     worker would too. Worth a focused investigation later.
- One operational tuning required: `bubble security set github
  write-graphql`. The default level
  (`allowlist-write-graphql`) blocks `gh repo view --json nameWithOwner`
  ("Field 'nameWithOwner' not in repository query allowlist") and
  `gh issue list --json …` queries that use top-level GraphQL aliases
  ("Aliases not allowed on top-level fields"). Per the threat model
  agreed at the start, accepting this trade-off is fine — it widens
  GraphQL to account-wide rather than repo-scoped, but the auth proxy
  still scopes REST and the bubble itself contains filesystem damage.
  Worth filing as a bubble allowlist-tuning issue if the project
  matures.

# 2026-05-04 05:56Z — feature session, issue #2011

## Accomplished

- Closed #2011 by adding `HexBerlekampZassenhaus/CrossCheck.lean`:
  LLL vs exhaustive recombination cross-check at lifted-factor counts
  `n ∈ {3, 4, 5, 6, 8}` with three fixtures per `n` (15 total).
- Each fixture polynomial is a product of distinct integer linear
  factors with `|root| ≤ 15`.  The `LiftData` is built directly with
  `p = 2`, `k = 8` (modulus 256) — the same shape used by the
  production-path branch guard `recombineLLLBranchGuardLift` already
  in `Basic.lean`.  Linear-factor coefficients fit inside the centred
  residue range `[-128, 127]`, so the fixtures are valid `mod p^k`
  representatives without invoking the full Hensel-lift pipeline (that
  surface is covered by `HexHensel/Conformance.lean`).
- Removed `private` from `recombineLLL?` in `Basic.lean` and added a
  docstring documenting the new cross-check use site, so the
  cross-check module can call the production LLL recombiner directly.
- Each `crossCheck` call asserts both algorithms return `some`, the
  factor multisets agree (sorted by negated constant term, which is
  the unique linear-factor signature for our distinct-root fixtures),
  and each algorithm's product reproduces the input polynomial.
- Verified the cross-check has independent value: a deliberate
  corruption to `recombinationSearch` (drop the last local factor)
  trips all five `#guard`s in `CrossCheck.lean` while leaving the
  existing `recombineLLLBranchGuardInput` guard in `Basic.lean`
  untripped — confirming the new module exercises a path not
  exclusively covered by existing tests.  Corruption reverted before
  commit.
- Wired `HexBerlekampZassenhaus.CrossCheck` into the library root
  `HexBerlekampZassenhaus.lean`.

## Current frontier

`HexBerlekampZassenhaus` now has a SPEC-sanctioned LLL-vs-exhaustive
cross-check at the boundary between the two recombination paths.  Note
the cross-check uses a directly-constructed `LiftData` rather than a
fresh trip through `henselLiftData`; the issue prefers the latter, but
on the obvious "product of distinct integer roots" fixtures
`choosePrimeData` falls back to its `p = 2` fallback (no admissible
prime survives the score-fold for these inputs), and the resulting
non-square-free `mod 2` image causes `multifactorLiftQuadratic` to
emit garbage lifted factors.  Re-routing the cross-check through
`henselLiftData` would require either a pre-tested polynomial whose
modular images are admissible at the smallest fallback-eligible prime,
or fixing the upstream `choosePrimeData` fallback.  Neither is in
scope for #2011.

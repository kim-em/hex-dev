# 2026-05-04T10:40Z `633f242d` (feature)

## Accomplished

Fixed `HexLLL` swap update producing non-`δ`-reduced output on
non-triangular integer bases (issue #2042).

The integer scaledCoeffs update for rows `i > k` in
`LLLState.swapStep` used the wrong formulas. The implementation read
matched the SPEC, but both were mathematically incorrect. The
ν[k][k-1] pivot, the d[k]' update, and the j < k-1 swap were all
fine; only the `i > k` two-column update was wrong.

**Wrong formulas (committed before this PR):**

    ν'[i][k-1] = (ν[i][k-1] · d[k]'  + ν[i][k] · B) / d[k]
    ν'[i][k]   = (ν[i][k]   · d[k-1] - ν[i][k-1] · B) / d[k]

**Correct formulas (Cohen Algorithm 2.6.3, derived from
basis(b')[k-1] = basis[k] + μ[k][k-1]·basis[k-1] and
μ'[i][k] = μ[i][k-1] − μ[k][k-1]·μ[i][k]):**

    ν'[i][k-1] = (d[k-1] · ν[i][k]   + B · ν[i][k-1]) / d[k]
    ν'[i][k]   = (d[k+1] · ν[i][k-1] - B · ν[i][k])   / d[k]

I.e. the prev-column factor is the *old* `d[k-1]` (not the *new*
`d[k]'`), and the curr-column factor is `d[k+1]` (not `d[k-1]`).

Numerically verified on the issue's reported failing input
`[[23,0,9,13],[13,10,-10,-4],[-12,14,15,8],[-8,-3,-13,21]]` (δ=3/4):

* old formula at first swap (k=1): ν'[2][0] numerator = -23,318,226,
  not divisible by d[k]=779 → silent integer truncation gives
  −29,933, swamping the rest of the run.
* new formula: numerator = −154,242 = 779·(−198), exact, gives
  ν'[2][0] = −198 = b[2]·b[1] as required by definition.

Existing conformance tests on `typical` (3×3) passed under the wrong
formula by coincidence: the wrong-formula numerator-pre-division was
`3` instead of `2`, and `3 / 2 = 1` (integer truncation) happened to
equal the correct quotient `2 / 2 = 1`.

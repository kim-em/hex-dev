# #5831 HO-5d-1b: partial executable-side migration off total choosePrimeData

Session UUID prefix: `4dc8257d`

## Accomplished

Three commits landed against the executable-side `choosePrimeData`
fallback discipline obligation (#5816 / #5831 / `SPEC/design-principles.md`
§8) in `HexBerlekampZassenhaus/Basic.lean`:

1. **`refactor: drop unused total toMonicPrimeData and related dead
   helpers`** — removes `ZPoly.toMonicPrimeData`,
   `ZPoly.toMonicLiftDataWithMonicPrime`, and the theorem
   `toMonicLiftDataWithMonicPrime_eq`. All three had no callers anywhere
   in the project (verified by `grep` across both
   `HexBerlekampZassenhaus/` and `HexBerlekampZassenhausMathlib/`).
   The Mathlib bridge already consumes the `?` variant
   `toMonicPrimeData?` directly. Eliminates one of the five total-
   `choosePrimeData` call sites flagged by PR #5830's audit.

2. **`refactor: migrate executable guard fixtures off total
   choosePrimeData`** — replaces `exhaustiveMonicCoreGuardPrimeData`
   (the `cldGuardF` lift expectation) and the
   `exhaustiveNonMonicQuadraticGuard` sign-normalization `#guard` so
   they dispatch on `choosePrimeData?` directly via the new
   `exhaustiveMonicCoreGuardPrimeData?` Option-returning helper.
   Both are private executable test fixtures with no proof-bridge
   coupling, so this is the safe wedge recommended by PR #5830's
   progress note. Eliminates two more call sites (private guards #2
   and #4 in the audit).

3. **`feat: add slow-path Option-propagating raw-factor variants`** —
   introduces `factorSlowFactorsWithBound? : ZPoly → Nat → Option
   (Array ZPoly)` and `exhaustiveSlowRawFactorsWithBound? : ZPoly →
   Nat → Option (Array ZPoly)` alongside the existing total versions,
   the slow-path counterparts of the already-`Option`-returning
   `factorFastFactorsWithBound`. Records the `some`-branch
   characterisation `factorSlowFactorsWithBound?_eq_some_iff_safe_branch`
   (mirrors `factorWithBound?_eq_some_iff_safe_branch` from PR #5830) and
   the equivalence `exhaustiveSlowRawFactorsWithBound?_eq_some_of_isSome`
   from the `choosePrimeData?.isSome` premise.

`lake build HexBerlekampZassenhaus` clean (143 jobs, only the
pre-existing single `sorry` warning at line 7204).
`python3 scripts/check_dag.py` succeeds.

## Current frontier

The remaining audit-flagged production callers
(`exhaustiveSlowRawFactorsWithBound`, the exhaustive arm of
`factorSlowFactorsWithBound`, and both arms of
`factorFastFactorsWithBound`) still consume the silent
`fallbackPrimeChoiceData` via total `choosePrimeData`. Migrating their
bodies in place requires updating ~111 downstream theorem references
that name `choosePrimeData` in their statements (e.g.
`factorSlowFactorsWithBound_branch`, `factorFastFactorsWithBound_branch`,
`exhaustiveSlowRawFactorsWithBound_mem_normalization_or_core`,
`factorWithBound_entry_mem_exhaustive_branch_*`, plus the
`*_polyProduct_of_some` proof chain). A trial `defeq` rewrite of
`exhaustiveSlowRawFactorsWithBound` to `(choosePrimeData? ...).getD
(fallbackPrimeChoiceData ...)` triggered `whnf` timeouts in
`exhaustiveSlowRawFactorsWithBound_mem_normalization_or_core` because
`rw [exhaustiveSlowRawFactorsWithBound]` produces a syntactically
different form from the theorem statement's `choosePrimeData`
reference.

`fallbackPrimeChoiceData` itself is still referenced in
`HexBerlekampZassenhaus/Basic.lean` (4 code hits: the definition at
line 1984, the total `choosePrimeData` `none` arm at line 2003, and
two proof witnesses on lines 2243 and 2257; the remaining hits are
docstring mentions introduced by this PR explaining the deferred
removal).

  or `audited-emergency-value` rationale, so this is the natural
  closer for the HO-5d arc.

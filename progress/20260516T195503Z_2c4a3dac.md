## Decomposition rationale

Two technical findings drove the split:

1. **`DensePoly.gcd` is not monic on monic inputs.** Concrete hand-
   worked counter-example over `ZMod 3`:

       gcd(x² − 1, x² + x) = 2x + 2

   Both inputs monic; output has leading coefficient `2`. The
   parent's proof outline routes through "each Berlekamp factor is
   monic" but the executable `splitFactorAt`'s underlying
   `DensePoly.gcd` does not preserve monicness. The `HexPolyFp`
   library exposes a separate `normalizeMonic` step that the
   square-free decomposition path uses, but Berlekamp's
   `splitFactorAt` does not currently apply it. Fixing this either
   means modifying `splitFactorAt` (an algorithm change with
   cascading proof updates on `prod_berlekampFactor` and the
   surrounding loop lemmas) or proving monicness via a different
   route — both belong in `HexBerlekamp/`, not in the Mathlib-
   bridge file.

2. **`splitFirstFactor?` can emit a unit cofactor.** When the
   kernel witness `w` satisfies `w − C c = u · fac` for a unit
   `u ≠ 1` (i.e., `fac` divides `w − C c` and the two have the
   same size), the Euclidean `gcd(fac, w − C c)` returns the
   accepted-by-`isNontrivialSplitFactor` value `u · fac`, and the
   cofactor `fac / (u · fac) = 1/u` lands in the running list as
   a degree-0 unit constant. So the naive "all factors have
   positive degree" loop invariant is false for the current
   algorithm, and `Nodup` maintenance over `splitFirstFactor?`
   needs structural reasoning that handles the unit-cofactor case
   (or modifies the algorithm to reject it).

The duplicate-implies-`isUnitPolynomial`-true direction is already
discharged by the existing
`isUnitPolynomial_of_squareFree_of_squared_dvd` in
`HexBerlekamp/RabinSoundness.lean`; the missing structural piece
is the loop invariant that rules out a `Nodup`-violating
duplicate in the first place.

## Current frontier

The committed bridge surface is binary-only because `HexGF2.GF2n`
packs `F_2` extensions into a single `UInt64` and there is no packed
counterpart for odd characteristic.  The issue body's `(p=3, n=2..6)`
and `(p=5, n=2..4)` size targets do not have a packed Lean rep to
bridge against; odd-characteristic `F_q` cross-checks live in
`HexGfqField`'s own oracle (#2003).

The `(p=2, n=4..16)` target is covered.  Higher binary degrees (e.g.
`n=32`) are easy to add by extending `HexGfq/EmitFixtures.lean` along
the existing per-`n` namespace pattern; the oracle handles arbitrary
`n` already.

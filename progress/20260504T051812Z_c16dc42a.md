# 2026-05-04 05:18Z — feature session, issue #2009

## Accomplished

- Closed #2009 by adding `HexGfq/CrossCheck.lean`: packed-vs-generic
  representation cross-check at extension degrees 4, 8, 16, 32 with
  100 pseudorandom shared inputs per degree per operation.
- Operations covered: addition, multiplication, inversion, Frobenius
  (`a ↦ a^2`). 16 `#guard`s total (4 ops × 4 degrees).
- Bridge encoding: each bit of a `UInt64` corresponds to one
  coefficient of the matching `FpPoly 2` representative, so the
  packed `GF2n` and generic `GFqField.FiniteField (FpPoly 2)` views
  are compared via a single `wordToPoly2` / `poly2ToWord` round-trip.
- Irreducibility witnesses for the ad-hoc moduli are `sorry`'d, in
  line with the existing pattern in `HexGF2/Conformance.lean` and
  `HexGF2/Bench.lean`.  The cross-check still runs operationally:
  each `#guard` actually evaluates both implementations and compares
  results.
- Wired the new module into `HexGfq.lean` so `lake build HexGfq`
  elaborates the cross-check (~14s for the CrossCheck file alone).
- Verified corruption-detection on both sides: corrupting
  `wordToPoly2` (generic side) trips all 16 `#guard`s; corrupting
  `(packedOf a + packedOf b).val` in N4 trips its `matchesAdd`
  guard.  Both deliberate corruptions reverted before commit.
- HexGfq covers `(p, n) = (2, 1)` already; this module exercises
  degrees beyond the committed Conway lookup.  Odd characteristic
  has no packed representation in HexGfq, so the cross-check is
  binary-only.

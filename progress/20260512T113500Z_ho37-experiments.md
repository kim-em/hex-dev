# HO-37 (#3693) — conformance experiments report

Five experiments to inform a structural fix for `conformance.yml`. Run
2026-05-12 over ~1 hour of work. Recommendations at the bottom; the
raw data is preserved per experiment so the recommendations can be
re-derived if assumptions shift.

## D — historical pass/fail on `main`

**Method**: pull last 1000 `Conformance` runs on `main` from the GH
Actions API (`gh run list --workflow Conformance --branch main --limit
1000`).

**Data** (window 2026-04-30 → 2026-05-12, ~12 days, 1000 runs):

| Conclusion  | Count | Notes |
|---|---|---|
| `cancelled` | 495   | Concurrency-cancel — superseded by next merge to main |
| `success`   | 262   | |
| `failure`   | 242   | |
| `?`         | 1     | (transient) |

**Among completed runs (504)**: **failure rate is 48% (242 / 504).**

**Failure-mode classification of 25 most recent failures** (`gh api
runs/<id>/jobs`, filter `conclusion == "failure"`):

| Failing component               | Count |
|---------------------------------|-------|
| `oracle-pyflint (HexGfq, ...)`  | 13    |
| `oracle-pyflint (HexBZ, ...)`   | 8     |
| `oracle-conway`                 | 5     |
| `conformance (HexGfq)`          | 5     |
| `conformance (HexGfqMathlib)`   | 5     |
| `conformance` (consolidated)    | 7     |
| Other                           | 2     |

Concentration: **HexGfq + HexBerlekampZassenhaus + Conway** account
for the overwhelming majority of recent failures. (Some runs are
multi-failure; sum > 25.) These are the libraries that were actively
under refactor; this is "engineering hot-spot," not "permanent risk
profile."

**Implication**: post-merge conformance does catch real regressions
(48% failure rate is substantial), but ~50% of all runs get
concurrency-cancelled before completing. Under high-cadence merging
the post-merge gate provides much less signal than its run count
suggests — many failures land on `main` and aren't flagged until a
later, slower merge window lets a run complete.

## C — cache-invalidation feasibility (local incremental rebuilds)

**Method**: from a warm local repo state, time per-target `lake build`
across the 27 + 14 conformance targets. Then make a cosmetic edit to a
high-fanout module (`HexPoly/Operations.lean`, imported by 24 files)
and rebuild.

**Per-target wallclock breakdown** (warm cache, this Mac):

| Target                          | Time   |
|---------------------------------|--------|
| HexGfqMathlib                   | 68 s   |
| hexgfqfield_emit_fixtures       | 86 s   |
| HexBerlekampZassenhausMathlib   | 50 s   |
| HexLLLMathlib                   | 46 s   |
| HexBerlekampMathlib             | 35 s   |
| HexConway                       | 35 s   |
| HexHenselMathlib                | 30 s   |
| HexGramSchmidtMathlib           | 23 s   |
| HexGF2Mathlib                   | 22 s   |
| HexPolyMathlib / HexPolyZMathlib / HexMatrixMathlib / HexModArithMathlib | 11-12 s each |
| All `Hex*` (non-Mathlib) computational libs | 1-5 s each |
| Most `*_emit_fixtures` exes     | 2-5 s each |

**Total wallclock (incremental from current state, this Mac)**: ~10
min. CI's clean baseline was ~25 min; the local warm test isolates
"which targets are inherently expensive to elaborate" from "fresh-
build cost."

The expensive targets cluster:

- 6 of the 11 Mathlib bridges contribute > 20 s each (≈350 s total).
- One emit-fixtures exe (`hexgfqfield_emit_fixtures`) is the single
  worst at 86 s.
- The 14 computational `Hex*` libraries are uniformly cheap (~30 s
  total combined).

**Cache-invalidation test**: appended `-- comment` to
`HexPoly/Operations.lean` (importer-fanout 24); ran `lake build $(conformance_targets)`.
Result: **2 modules rebuilt** (`HexPoly.Operations` itself, plus
`HexArith.Conformance` from leftover prior-step cache), **16 modules
"Replayed"** from cache, total wallclock 12 s.

This confirms Lake's incremental build is content-hash-aware at the
elaborated-module level: cosmetic source changes (whitespace,
comments) don't propagate to dependents, only changes that affect the
elaborated output do. A GH Actions cache on `.lake/build/lib/lean/Hex*`
is therefore viable — restoration + `lake build` will skip work for
all modules whose elaborated content is unchanged, and recompile only
genuinely-affected dependents.

## A — per-target compile breakdown on CI (clean cache)

**Method**: branch `experiment/conformance-timings` with per-target
timing instrumentation; `gh workflow run conformance.yml --ref
experiment/conformance-timings`. First run (`25730688312`) hit the
6-hour GH Actions wall-clock cancel during the oracle step and the
log persistence failed. Second run (`25750515799`) added `if: false`
to the oracle step so only the build phase ran, completed in 34 min
total. Build phase wallclock: **31m51s** (consistent with the
unsplit baseline's ~30 min).

**Per-target CI wallclock** (clean .lake/build, Mathlib cache
restored, ubuntu-latest runner):

| target                          | sec   | role                                          |
|---------------------------------|-------|-----------------------------------------------|
| **HexBerlekampMathlib**         | **740** | Mathlib bridge — single biggest item, 12 min |
| **HexGF2**                      | **399** | computational lib — 6.7 min                  |
| **hexgfqfield_emit_fixtures**   | **163** | emit-fixture exe — 2.7 min                   |
| HexBerlekamp                    | 145   | computational lib                              |
| HexGfqMathlib                   | 82    | bridge                                         |
| HexConway                       | 77    | computational lib                              |
| HexBerlekampZassenhaus          | 54    | computational lib                              |
| HexLLLMathlib                   | 41    | bridge                                         |
| HexArith                        | 41    | computational lib                              |
| HexGF2Mathlib                   | 30    | bridge                                         |
| HexBerlekampZassenhausMathlib   | 28    | bridge                                         |
| (rest, each < 20 s)             |       |                                                |
| **TOTAL**                       | **1911 (31m51s)** |                                    |

**Concentration**: the top 4 targets (HexBerlekampMathlib, HexGF2,
hexgfqfield_emit_fixtures, HexBerlekamp) account for **1447 s =
76 %** of the total wallclock. Even the top 2 alone (HexBerlekampMathlib
+ HexGF2) consume **60 %** of the build.

**Bridges contribute** 977 s combined (51 % of total):
HexBerlekampMathlib (740) + HexGfqMathlib (82) + HexLLLMathlib (41) +
HexGF2Mathlib (30) + HexBerlekampZassenhausMathlib (28) +
HexGramSchmidtMathlib (15) + HexHenselMathlib (14) + HexPolyZMathlib +
HexModArithMathlib + HexMatrixMathlib (each 7) + HexPolyMathlib (6).

**Surprise**: HexBerlekampMathlib is wildly out of proportion with
other bridges. It elaborates substantially harder than its siblings.
Worth a separate look — possibly a single heavy proof or import-graph
explosion specific to that module.

**Surprise 2**: HexGF2 is the slowest non-bridge at 399 s — 6.7 min
for a computational library is high. Probably the cumulative
elaboration of its many sub-modules (Clmul, Multiply, Euclid, Field,
Irreducibility, CommonIrreducibility, RabinSoundness, etc., per the
GF2-related builds in the bench-exe pre-build step).

## Smoke-variant projected wallclock (refined with A's data)

Computing the smoke = "drop the 11 Mathlib bridges" projection
against the real CI numbers:

  full conformance  : 1911 s
  bridges removed   :  977 s
  smoke wallclock   :  934 s = 15m34s

That's **a ~51 % reduction** but still **> 10 min**, the HO-37
verification target. To clear the 10-min bar we'd need to either:

- also cache `.lake/build` (C-confirmed viable; would save most of
  the remaining 934 s on cache hits)
- separately attack HexGF2 (399 s) and hexgfqfield_emit_fixtures
  (163 s) — but those are real computational targets we don't want
  to drop from the gate.

## Recommendations (updated with A data)

In rough order of effort × payoff:

1. **Drop the 11 Mathlib bridges from conformance** (E + B + A). The
   single biggest win: removes 977 s (51 %) of the build, and the
   biggest offender (HexBerlekampMathlib at 740 s alone). The
   bridges still compile under `ci.yml`'s `lake build` — we just
   stop paying for them in the conformance gate. Run them in a
   nightly `--full` variant if we want the proof-side coverage
   surfaced separately.

2. **Add `.lake/build` GH Actions cache to the (now-smoked)
   conformance gate.** Cache key: `lean-toolchain` +
   `lake-manifest.json` + content hash of `Hex*/**.lean`. Local C
   confirmed Lake's incremental is content-aware, so on a cache hit
   touching N files, only ~N modules + their semantic dependents
   rebuild. Expected effect on a typical small-diff PR: 934 s →
   30-90 s.

3. **Re-enable `pull_request` trigger on the smoke variant only**.
   Re-add the smoke check to `main` branch protection's required
   contexts after observing at least one PR cycle where the smoke
   gate fires green in < 10 min.

4. **Nightly schedule** in `conformance.yml` for the full
   `--full` variant (including bridges + the slow outliers). No
   per-PR latency, no concurrency-cancel pressure because nightly
   runs are spaced.

5. **Profile HexBerlekampMathlib's 740 s** as a follow-up bug. Even
   under nightly, 12 min for a single library is suspicious; a
   single heavy proof or import-graph regression likely accounts
   for most of it.

6. **Profile HexGF2's 399 s** similarly. Real computational
   substance there but 6.7 min is high; some of those sub-modules
   may be splittable or have macro-elaboration hot-spots.

## E — smoke-variant viability

**Method**: synthesise from B + D. Question: would a "smoke" gate
(fast subset of conformance, runnable per-PR) catch the historical
failure pattern?

**Proposed smoke subset**: the 14 oracle-having libraries only — drop
the 11 Mathlib bridges (977 s of CI compile time per A, zero oracle
coverage) and the 2 computational-no-oracle libs (HexArith,
HexModArith — also nothing to cross-check).

**Coverage check against D's 25-failure sample**:

| Failure pattern                          | Caught by smoke? |
|------------------------------------------|------------------|
| `oracle-pyflint (HexGfq, ...)` × 13      | YES (HexGfq is in the 14) |
| `oracle-pyflint (HexBZ, ...)` × 8        | YES |
| `oracle-conway` × 5                      | YES |
| `oracle-pyflint (HexBerlekamp, ...)` × 1 | YES |
| `conformance (HexGfq)` × 5               | YES |
| `conformance (HexGfqMathlib)` × 5        | **NO** (Mathlib bridge skipped) |
| `conformance` (consolidated, recent) × 7 | YES (build-step compile) |
| `Install system deps` × 1                | YES (still in smoke) |
| `Set up Lean toolchain` × 1              | YES |

**Coverage**: the smoke subset would catch ~92 % of the historical
failures in this sample. The misses are the 5 `HexGfqMathlib`
proof-side regressions — but those were specific to the active
HexGfqMathlib refactor; they don't represent a permanent risk.

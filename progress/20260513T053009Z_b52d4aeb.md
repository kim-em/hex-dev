# HO-39: strip ceremonial `Hex*Mathlib/Conformance.lean` files

## Accomplished

- Deleted the 11 ceremonial `Hex*Mathlib/Conformance.lean` files
  (`HexBerlekampMathlib`, `HexBerlekampZassenhausMathlib`,
  `HexGF2Mathlib`, `HexGfqMathlib`, `HexGramSchmidtMathlib`,
  `HexHenselMathlib`, `HexLLLMathlib`, `HexMatrixMathlib`,
  `HexModArithMathlib`, `HexPolyMathlib`, `HexPolyZMathlib`).
- Removed the corresponding `import …Mathlib.Conformance` lines from
  each root `Hex*Mathlib.lean`. For `HexLLLMathlib.lean` the only
  surviving import was the Conformance module, so I swapped it for the
  obvious `import HexLLLMathlib.Basic`.
- Audited every `#guard` and `#eval` in the deleted files against the
  computational sibling `Hex*/Conformance.lean` modules. Each bridge
  `#guard` either exercises the bridge function itself (no Hex
  executable to migrate) or duplicates coverage that the sibling
  already provides:
  - `HexBerlekamp/Conformance.lean` already covers `rabinTest` /
    `checkIrreducibilityCertificate`.
  - `HexBerlekampZassenhaus/Conformance.lean` already covers
    `Factorization.product = input` across the `factorizationEdgeCases`
    list (zero, one, constants, `X`, products of linears, etc.).
  - `HexMatrix/Conformance.lean` already covers `Matrix.row` and
    `Matrix.rowCombination`.
  - `HexGF2/Conformance.lean` already covers `GF2Poly.toWords`,
    `ofUInt64`, `monomial`, etc.
  Bridge tests (e.g. `castIntMatrix`, `toZMod`/`ofZMod`,
  `FpPoly.coeffIndex`, `FpPoly.ofIndexBelowDegree`) are not Hex
  executable code and the SPEC's new banned anti-patterns rule out a
  bridge `Conformance.lean` for them.

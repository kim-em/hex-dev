# HexModArith Prime API Phase 6 polish

## Accomplished

Reviewed `HexModArith/Prime.lean` for Phase 6 API quality and added the two
symmetric-inverse wrappers called out by issue #3572.

- `mul_inv_eq_one_of_prime`: explicit-primality form, derived from
  `inv_mul_eq_one_of_prime` via `grind` (commutativity from the existing
  `Lean.Grind.CommRing (ZMod64 p)` instance).
- `mul_inv_eq_one_of_ne_zero`: `[PrimeModulus p]` class-based variant, matching
  the parallel structure of `inv_mul_eq_one_of_ne_zero`.

Audited downstream consumers (`HexBerlekamp`, `HexPolyFp`, `HexGfqField`,
`HexBerlekampZassenhaus`, `HexHenselMathlib`). The only repeated pattern that
was actively re-deriving an inverse equality is the private helper
`zmod64_mul_inv_eq_one_of_prime_ne_zero` in `HexPolyFp/SquareFree.lean`
(five call sites), which can now be replaced by the new public wrappers in a
follow-up. Other patterns
(`(ZMod64.PrimeModulus.prime (p := p)).two_le`,
`Hex.Nat.Prime.two_le (ZMod64.PrimeModulus.prime ...)`) are already one-liner
projections off the existing class accessor; no additional wrapper was
warranted within the Phase 6 "at most two" budget.

Naming follows the existing `inv_mul_eq_one_of_prime` /
`inv_mul_eq_one_of_ne_zero` pair: `mul_inv_*` is the symmetric form, the
`_of_prime` suffix indicates explicit primality, `_of_ne_zero` indicates the
class-instance variant.

## Current frontier

Phase 6 polish on `HexModArith/Prime.lean` is at a natural stopping point.
`HexModArith/Ring.lean` was inspected for naming consistency only; no edits
required.

## Why deliverable 2 deviates from the issue's illustrative signature

The issue suggests:

```lean
theorem choosePrimeData_leadingCoeff_castRingHom_ne_zero
    (f : Hex.ZPoly) (hpos : 0 < f.size)
    (hcore_pos : 0 < (Hex.normalizeForFactor f).squareFreeCore.size) :
    (Int.castRingHom
        (ZMod (Hex.choosePrimeData (Hex.normalizeForFactor f).squareFreeCore).p))
      ...
      ‚Č† 0
```

with unconditional `choosePrimeData`. But `choosePrimeData`
(`HexBerlekampZassenhaus/Basic.lean:1516`) calls back to
`fallbackPrimeChoiceData` when no candidate in the small prime list
satisfies `isGoodPrime`. The fallback hard-codes `p = 3` and does not
re-check goodness, so for a core whose leading coefficient is divisible
by 3 and whose small-prime list is otherwise unfit, the cast leading
coefficient is zero. Even with `hpos` / `hcore_pos`, the unconditional
statement is provably false on those inputs.

The issue's own escape clause ‚ÄĒ "If the positivity hypotheses are
unnecessary ... drop them; the signature is illustrative" ‚ÄĒ explicitly
authorises substituting a tighter signature. I used the existing
`_of_choosePrimeData?` wrapper pattern (cf.
`squareFreeCore_irreducible_of_small_mod_singleton_of_choosePrimeData_squareFreeModP`
at line 558) with `hselected : choosePrimeData? core = some primeData`.
The `isGoodPrime` form is also exposed for callers that hold the
goodness invariant by other means.

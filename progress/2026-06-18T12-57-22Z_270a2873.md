## Accomplished

Closed the fourth `modulusFpPoly` sorry from #7915:
`HexGF2Mathlib.GF2nPoly.modulusFpPoly_degree_pos` in
`HexGF2Mathlib/Field.lean`.

The directive premise ("an irreducible `GF2Poly` is non-constant") is false
for the project-local `Hex.GF2Poly.Irreducible`, which admits the unit `f = 1`
vacuously. The theorem is therefore unprovable from `hirr` alone (the #7919
docstring already documented this).

Sound fix chosen: add the positivity as an explicit hypothesis
`hdeg : 0 < f.degree`, mirroring exactly the single-word `GF2n` sibling, which
already carries `hn : 0 < n` and derives its modulus degree from it. This is a
strictly smaller change than the two options sketched in the issue: it touches
only the Mathlib bridge file (`HexGF2Mathlib/Field.lean`), leaves the
executable `Hex.GF2Poly.Irreducible`/`GF2nPoly` structure untouched (no risk to
the merge-gated executable build / conformance / bench), and needs no new
infrastructure. The proof is `degree?_toFpPoly` + the hypothesis.

`hdeg` is threaded (named-arg, implicit, like `hn`) through the
`GenericFiniteField` cluster: `modulusFpPoly_degree_pos`, `GenericFiniteField`,
`toGeneric`, `ofGeneric`, `ofGeneric_toGeneric`, `toGeneric_ofGeneric`,
`toGeneric_add`, `toGeneric_mul`, `equiv`. The cardinality side
(`finEquiv`/`fintype_card`) is independent of `GenericFiniteField` and was left
alone. No external consumers of the `GF2nPoly` Mathlib API exist.

`lake build HexGF2Mathlib` is green (1131 jobs). Diff is sorry/axiom-free.

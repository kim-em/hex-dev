## Decision: the ring-axiom lemmas stay `@[simp]`-only

Deliverable 2 asked to re-check `add_zero`, `zero_add`, `mul_zero`,
`zero_mul`, `mul_one`, `one_mul`, `pow_zero`, `pow_one`, `pow_succ`. The
file establishes `Lean.Grind.Semiring`, `Lean.Grind.Ring`, and
`Lean.Grind.CommRing (ZMod64 p)` instances, so grind already has a
commutative-ring procedure for this type.

Empirical probe (temporary `example ... := by grind`, no explicit
lemma list, removed before commit): grind closes **all nine**
ring-axiom goals — `a + 0 = a`, `a * 1 = a`, `a ^ (n+1) = a^n * a`,
etc. — directly from the `CommRing` instance. Promoting them to
`grind =` would duplicate the built-in ring procedure's rules, so they
are left `@[simp]`-only.

The same probe showed grind does **not** close the variable-exponent
powers `(0 : ZMod64 p) ^ n = 0` (`n ≠ 0`) and `(1 : ZMod64 p) ^ n = 1`
from the instance (the ring procedure handles literal exponents only).
These are the genuinely useful promotions: a second probe confirmed
both now close via the new `grind =` rules, with no loop. `zero_pow`'s
conditional `n ≠ 0` form is accepted cleanly by `grind =`.

This matches the contrast with sibling sweeps: `HexPolyFp/Quotient.lean`
and `HexPolyFp/Basic.lean` promoted `add_zero`/`mul_one`-style lemmas
because those types carry *no* grind ring instance, so grind cannot
derive them.
